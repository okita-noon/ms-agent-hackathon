# Agent動作設計・テスト戦略

> AI Agent の自然な返信を活かしつつ、受注確定・在庫引当・要対応判断は業務ルールとして確実に守るための設計。

## 狙い

食品卸の受注では、返信文が自然であることよりも「確定してよい注文だけを確定する」ことが重要になる。
そのため、このアプリケーションでは AI Agent の生成結果をそのまま業務確定に使わず、以下を分離する。

| 領域 | 方針 |
|---|---|
| 対話・解釈 | AI Agent が注文文、会話履歴、省略表現を解釈する |
| 業務状態 | コード側で `order_id` / `review_order_id` / `session_status` を決める |
| 顧客返信 | Agent が自然文を生成し、最後に返信ポリシーで安全性を検査する |
| ダッシュボード | `要対応` は受注合計から分離し、担当者確認対象として表示する |

## Agentの責務分離

```text
LINE / 電話 / メール
  → Intake Agent: 顧客・商品・数量・納品条件を構造化
  → Exception Agent: 数量・単位の異常を検知
  → Inventory Agent: 在庫確認・引当・代替候補
  → OrderOrchestrator:
       - 確定してよい場合だけ order_id を作成
       - 在庫不足・引当不可は review_order_id として保存
       - 確認が必要な場合は awaiting_reply にする
  → Communication / Orchestrator Agent: 顧客向け返信を生成
  → Response Policy Guard: 未確定時の確定表現を禁止
```

## 確定・要対応・確認待ちの判定

| 状態 | 保存先・戻り値 | 顧客への伝え方 | ダッシュボード |
|---|---|---|---|
| 在庫引当まで完了 | `order_id` | 受注済みとして返信 | 受注件数・受注合計に含める |
| 在庫不足・引当不可・代替あり | `review_order_id` | 担当者確認中と返信 | `要対応` として表示し、受注合計から除外 |
| 数量・単位など顧客確認が必要 | `session_status=awaiting_reply` | 確認質問を返信 | 受注確定しない |
| 解析不能 | `response` のみ | 追加情報を依頼 | 受注確定しない |

この分離により、LLM の返信文が揺れても、在庫不足の注文を誤って受注済みにしない。

## LINE会話履歴の使い方

LINE は「OK」「それで」「1個で」のような短い返信が多い。
そのため、Webhook 受信時に以下を `OrderOrchestrator` へ渡す。

- 直近20件の `message-history`
- active / awaiting_reply session の `pending_order_draft`
- 現在のユーザーメッセージ

明確な肯定返信と `pending_order_draft` がある場合は、Intake Agent に再解釈させず、保存済みドラフトから直接受注確定する。
省略表現や変更指示の場合は、会話履歴をプロンプトに含めて Agent が文脈を参照する。

## テスト戦略

AI の返信文を完全一致でテストすると、自然文の揺れに弱くなる。
このプロジェクトでは、以下の3層で判定する。

| レイヤー | 判定内容 | 例 |
|---|---|---|
| Deterministic test | 業務状態が正しいか | 在庫不足時に `order_id` を作らず `review_order_id` を作る |
| Response policy test | 未確定時に危険な表現をしていないか | `受注しました` / `確定しました` / `承りました` を禁止 |
| Scenario test | 会話履歴・確認待ちドラフトが渡るか | 「OK」で pending draft が確定される |

AI Judge は、返信の自然さ・わかりやすさ・顧客体験の評価には有効。
一方で、受注確定可否や在庫不足時の処理など、業務上の合否はコードで決まるルールベーステストを優先する。
AI Judge を使う場合も、初期は CI の必須条件ではなく warning / レビュー補助として扱う。

## 実装済みガード

| ファイル | 内容 |
|---|---|
| `src/agents/orchestrator.py` | 在庫不足・確認待ちの判定、`review_order_id` 保存、返信ポリシーガード |
| `tests/test_orchestrator.py` | 在庫不足時の要対応保存、未確定時の禁止表現、会話履歴コンテキスト |
| `tests/test_line_handler.py` | LINE履歴保存、pending draft 引き継ぎ、重複返信防止 |

## Zenn記事で拾うポイント

記事では、設計を単独で説明するよりも、以下のように価値へ接続して書く。

```text
AI Agent の返信文そのものを完全一致でテストするのではなく、
「受注確定してよい状態か」「要対応にすべきか」
「在庫不足時に確定表現を使っていないか」をルールとして検証しています。

これにより、AI の柔軟な対話能力を活かしながら、
業務上事故につながる受注確定だけは deterministic に制御しています。
```

この構成は、Microsoft Agent Hackathon 2026 の審査観点である Agentic AI としての有効性、
アーキテクチャの妥当性、安定動作・運用性、ビジネスインパクトを説明しやすい。
