# デバッグログガイド

> OrderOrchestrator の処理過程を可視化するデバッグログの仕様と解析方法。

## 概要

LINE Tester（`/line-tester`）からメッセージを送信すると、OrderOrchestrator の各処理ステップがデバッグログとして記録される。
ログは API レスポンスの `debug_log` フィールド（`list[str]`）で返却され、LINE Tester の UI 上で色分け表示される。

## ログの流れ

```text
メッセージ受信
  → [入力] / [履歴] / [現在注文]
  → [判定] アクション分類（肯定返答・取消・在庫問い合わせ etc.）
  → [Intake] 注文解析
  → [在庫] 在庫チェック
  → [配送] 配送日推定
  → [保存] 受注保存 or ドラフト保存
  → [応答生成] or [Communication] 返信生成
  → [応答] 最終返信
```

## タグ一覧

### 入力・コンテキスト

| タグ | 内容 | 例 |
|---|---|---|
| `[入力]` | 受信メッセージ、チャネル、セッションID | `[入力] source=line, session=S-xxx, message='りんご10箱'` |
| `[履歴]` | 会話履歴の件数 | `[履歴] 5件の会話履歴あり` |
| `[現在注文]` | アクティブな注文がある場合 | `[現在注文] ORD-xxx (status=pending)` |

### 判定・分岐

| タグ | 内容 | 出現条件 |
|---|---|---|
| `[判定]` | オーケストレータの分岐判定結果 | 肯定返答検出、ドラフト有無、取消判定時 |
| `[取消]` | 注文キャンセル実行 | ユーザーが取消を指示した場合 |

### Agent 処理

| タグ | 内容 | 例 |
|---|---|---|
| `[Intake]` | Intake Agent の処理結果 | プロンプト種別、応答文字数、処理時間、生テキスト冒頭 |
| `[Communication]` | Communication Agent の処理結果 | LLM入力要素、応答時間・文字数 |

### 業務処理

| タグ | 内容 | 例 |
|---|---|---|
| `[在庫]` | 在庫チェック結果 | `[在庫] 全品在庫OK (3件チェック)` |
| `[配送]` | 配送日推定の入出力 | `[配送] 入力: route=北関東便, lead_time=翌日` / `[配送] 推定配送日: 2026-05-28` |
| `[保存]` | 受注・ドラフトの保存結果 | 受注確定ID、ドラフト内容、要対応保存、保存失敗 |

### 応答生成

| タグ | 内容 | 例 |
|---|---|---|
| `[応答生成]` | Orchestrator Agent による返信生成 | LLM入力要素、処理時間・文字数 |
| `[応答]` | 最終応答の情報 | 文字数、LINEテンプレート上書き有無 |

## UI での表示

LINE Tester（`/line-tester`）の各メッセージに「Debug」トグルボタンがある。
クリックするとダークテーマのログパネルが展開され、タグ部分が青色、メッセージ部分がオレンジ色で表示される。

## 解析の手引き

### Agent の応答が遅い場合

`[Intake]`・`[Communication]`・`[応答生成]` の秒数（`Xs`）を確認する。
どのAgent呼び出しがボトルネックか特定できる。

### 注文が確定されない場合

1. `[Intake]` の生テキスト冒頭で、Agent が注文内容を正しく解析しているか確認
2. `[在庫]` で在庫不足が出ていないか確認
3. `[保存]` で「要対応注文として保存」になっていないか確認
4. `[判定]` でドラフト→肯定返答の流れが正しく動いているか確認

### 返信内容がおかしい場合

1. `[Communication]` または `[応答生成]` の「LLM入力要素」を確認し、必要な情報が渡っているか見る
2. `[応答]` で「LINEテンプレート上書き」が発生していないか確認（テンプレートが意図しない返信を上書きしている可能性）

### 配送日がおかしい場合

`[配送]` の入力（route・lead_time）と出力（推定配送日）を確認する。
顧客マスタの `delivery_lead_time` やカレンダー設定（定休日・臨時休業）が原因の可能性がある。
