# 業務ドメイン定義

> 実際の業務データから抽出した、システムが扱うドメイン概念の定義。

## 受注ライフサイクル

```
受注 → 手配 → 製造 → ピッキング → 配送 → 完了
```

### ステータス遷移

| ステータス | 意味 | 次のステータス |
|---|---|---|
| **未処理** | 注文を受信したが手配未着手 | 製造 / 完了 / キャンセル / 返信待ち |
| **返信待ち** | 確認質問を送信し顧客返信待ち | 未処理（返信受信後に再処理） |
| **製造** | 手配済み。商品の準備・製造中 | 配送 |
| **配送** | ピッキング完了。配送中 | 完了 |
| **完了** | 納品済み | — |
| **キャンセル** | 注文取消 | — |

### 3つの日付

| 日付 | 意味 | 例 |
|---|---|---|
| **受注日** | 注文を受けた日 | 7/14 |
| **手配日** | 準備・製造を開始する日（通常は受注日の翌営業日） | 7/15 |
| **納品日** | 顧客に届ける日（手配日の翌日〜） | 7/16 |

## 受注チャネル

実業務では FAX・ECサイト も主要チャネルだが、**MVP では LINE のみ実装**する。
データモデル上は全チャネルを定義済みで、将来の拡張に対応。

| チャネル | `source` 値 | `system` 値 | MVP対象 | 備考 |
|---|---|---|---|---|
| LINE | `LINE` | `LINE` | **対象** | Webhook 受信。MVPの主要チャネル |
| FAX | `FAX` | `FAX` | 対象外 | 実業務では主要だがMVPでは未実装 |
| ECサイト | `Web` | `ECサイト` | 対象外 | 実業務では主要だがMVPでは未実装 |
| メール | `Email` | `Email` | 対象外 | Graph API で受信 |
| 電話 | `Phone` | `Phone` | 対象外 | ACS + AI Speech で文字起こし |
| 手入力 | `手入力` | — | **対象** | 担当者がダッシュボードから直接入力 |

## 温度帯管理

食品物流の基本概念。商品ごとに固定で、配送便の選択に直結する。

| 温度帯 | 保管温度 | 配送手段 |
|---|---|---|
| **常温** | 15〜25°C | 自社便 / 芦川便 |
| **冷蔵** | 0〜10°C | 自社便 / 芦川便 / 冷蔵ヤマト便 |
| **冷凍** | -18°C以下 | 自社便 / 冷凍ヤマト便 |

## 配送体系

### 配送便（Carrier）

| 便名 | 種類 | 温度帯 | 備考 |
|---|---|---|---|
| **自社便** | 自社トラック | 全温度帯 | 近距離・定期ルート |
| **芦川便** | 委託配送 | 常温・冷蔵 | 遠距離・一部ルート |
| **冷蔵ヤマト便** | ヤマト運輸 | 冷蔵 | 個別配送。送り状番号あり |
| **冷凍ヤマト便** | ヤマト運輸 | 冷凍 | 個別配送。送り状番号あり |

## 不定貫（ふていかん）

食品業界特有の概念。**重量が一定でない商品**（果物・肉など天然物）。

- 受注時: 数量は「箱」「個」で注文（例: もも 8箱）
- ピッキング時: 実際に計量して重量確定
- 請求時: 実測重量 × 単価 で計算

### データ上の扱い

- `is_variable_weight = true` の商品は不定貫
- 受注時の `quantity` は概算（空の場合もある）
- ピッキング後に `actual_weight` を記録

## ピッキングリスト

受注データからピッキングリスト（出荷指示書）を生成するロジック:

1. **納品日でグループ化**: 同じ納品日の受注をまとめる
2. **配送便で分割**: 温度帯・配送先に応じて便を割り当て
3. **1受注 → 複数ピッキング行**: 遠距離や温度帯混載の場合、1つの受注が複数便に分割される
