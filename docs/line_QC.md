## 1. 目的
LINE Tester (`/line-tester`) の品質管理ドキュメント。  
**テストケース母集団**と**実施ログ**を1ファイルに集約する。

## 2. 運用ルール
1. 1テストは3ケースで回す。
2. ログはこの `line_QC.md` に追記する（ファイル分割しない）。
3. 回次は `4.1` -> `4.2` -> `4.3` のように増やす。
4. 各回に以下を残す。  
   4.1. 実施日時（JST）  
   4.2. 参照ブランチ / mainの最大PR番号  
   4.3. 顧客No  
   4.4. 「今の状態+応答メッセージ」  
   4.5. 所感 / 理想応答 / 対策

### 3.1. ケース一覧（固定）
`CLAUDE.kio.md` の「#6. LINE動作チェック用テストケース集」を共通ケースとして移管。

#### 4.1.1. Test1: 新規注文 -> 修正 -> 確定表現
1. 入力: `りんご10箱、バナナ20kgお願いします`
   - 応答: 在庫0で受付不可（りんご/バナナ）
   - 今の状態+応答メッセージ: `order_saved=false / order_id=null / current_order_id=null` + `りんごは在庫が0箱のため10箱の注文を受け付けられません。バナナは在庫が0kgのため20kgの注文を受け付けられません。ご了承ください。`
2. 入力: `やっぱり、りんご5箱でお願いします`
   - 応答: 現在注文なしの確認文
   - 今の状態+応答メッセージ: `order_saved=false / order_id=null / current_order_id=null` + `確認です。変更対象の現在注文が見当たりませんでした。新しいご注文として承る場合は、商品名と数量をあらためてお知らせください。`
3. 入力: `それでお願いします`
   - 応答: 在庫0で受付不可（りんご5箱）
   - 今の状態+応答メッセージ: `order_saved=false / order_id=null / current_order_id=null` + `りんごは在庫が0箱のため5箱の注文を受け付けられません。ご了承ください。`

所感:
1. 会話として破綻はしていないが、在庫0判定で新規注文文脈が切れやすい。
2. `debug_log` がAPI応答に含まれず、UI側デバッグ活用が難しい。

理想応答（例）:
1. 欠品時に代替行動を提示する。
2. 例: `りんごは現在欠品です。数量変更・代替商品・次回入荷確認のいずれをご希望ですか？`

#### 4.1.2. Test2: 注文 -> 曖昧修正 -> 再指定
1. 入力: `卵を3箱お願いします`
   - 応答: 在庫0で受付不可
   - 今の状態+応答メッセージ: `order_saved=false / order_id=null / current_order_id=null` + `卵は在庫が0箱のため3箱の注文を受け付けられません。ご了承ください。`
2. 入力: `さっきのやつ減らして`
   - 応答: 現在注文なしの確認文
   - 今の状態+応答メッセージ: `order_saved=false / order_id=null / current_order_id=null` + `確認です。変更対象の現在注文が見当たりませんでした。新しいご注文として承る場合は、商品名と数量をあらためてお知らせください。`
3. 入力: `卵を1箱でお願いします`
   - 応答: **`不明は在庫が0箱...`**
   - 今の状態+応答メッセージ: `order_saved=false / order_id=null / current_order_id=null` + `不明は在庫が0箱のため1箱の注文を受け付けられません。ご了承ください。`

所感:
1. 3通目の `不明` 表示は不自然。

理想応答（例）:
1. `卵は在庫が0箱のため1箱の注文を受け付けられません。`
2. 「不明」をユーザー向け文面に出さない。

対策:
1. 在庫ステータスが `不明` のとき、抽出商品名へフォールバックする修正を実装。
2. ユニットテスト追加（`tests/test_orchestrator.py`）。

#### 4.1.3. Test3: 在庫問い合わせ -> 条件付き注文 -> 納品条件
1. 入力: `白菜ある？`
   - 応答: 商品名再入力要求
   - 今の状態+応答メッセージ: `order_saved=false / order_id=null / current_order_id=null` + `確認したい商品名をもう一度教えてください。`
2. 入力: `あれば2ケースお願いします`
   - 応答: 商品特定不可メッセージ
   - 今の状態+応答メッセージ: `order_saved=false / order_id=null / current_order_id=null` + `「白菜」についてのご注文ですね。こちらでは商品特定できなかったため、正式な商品名や規格（例: 1玉、1箱など）を教えてください。`
3. 入力: `明日午前中で`
   - 応答: 白菜2ケース在庫0
   - 今の状態+応答メッセージ: `order_saved=false / order_id=null / current_order_id=null / pending_exists=true` + `白菜は在庫が0ケースのため2ケースの注文を受け付けられません。ご了承ください。`

所感:
1. 文脈保持は一定効くが、1通目で「白菜」照会に入れず改善余地あり。

理想応答（例）:
1. 1通目で `白菜の在庫を確認します` と返す。
2. 2通目で在庫数と可否を明示する。
