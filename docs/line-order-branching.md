# LINE受発注の分岐・会話テンプレート・要件整理

> 飲食店と食品卸の間で LINE 受発注を運用する際の主要分岐を、業務フロー・会話例・システム要件に落とし込んだ整理。

## 1. この文書の目的

この文書は、以下の3点を一度に扱う。

1. **分岐の見取り図**: 何が起きるとどこで分かれるか
2. **LINE 会話テンプレート**: 顧客と卸のやり取りをどう返すか
3. **システム要件整理**: どの分岐をどう判定し、何を保存するか

対象は、**飲食店 → 食品卸** の日常受発注である。
LINE は短文・省略・追記・修正が多いため、単なる注文抽出ではなく
**会話継続** と **現在注文の更新管理** を前提に設計する。

## 2. 重要な業務前提

今回の業務前提は、一般的な EC の複数同時注文とは異なる。

1. **1顧客につき、配送完了まで開いている注文は1件**  
   顧客ごとに複数注文が並行して走る想定は置かない。
2. **配送完了前に届く LINE は、基本的に現在注文への追記・変更・取消依頼として扱う**
3. **現在注文がある間は、新しい注文を増やすより先に現在注文の差分として解釈する**  
   たとえば `白菜も追加で` `卵だけなしで` `さっきのトマトを15kgで` のような短文は、
   原則として現在注文への差分操作として扱う。
4. **LINE では受注Noを表示しない**  
   受注Noは内部管理用とし、顧客には見せない。
5. **配送中・出荷確定後の変更は、自動反映せず担当者確認へ送る**

この前提により、LINE の主役は **新規注文作成** だけでなく
**現在注文の更新** になる。

### 2.1. MVP の運用ルール

MVP では、判断を次のように単純化する。

| 状況 | 扱い |
|---|---|
| 現在注文なし | 新規注文として受ける |
| 現在注文あり + 注文系メッセージ | その現在注文への更新依頼として扱う |
| 現在注文あり + 編集不可 | 自動変更せず `要対応` に送る |
| 納品完了後 | 次回メッセージは新規注文として扱う |

つまり、**同じ顧客から配送前に来る注文系 LINE は、別注文ではなく同一注文の更新** とみなす。

## 3. 全体フローチャート

### 3.1. 受信直後の大分岐

```text
LINE メッセージ受信
  -> 1. 顧客を特定する
       -> 特定できる
       -> 特定できない -> 本人確認 / 担当者対応

  -> 2. 顧客の現在注文を探す
       -> 現在注文なし
            -> 新規注文候補として扱う
       -> 現在注文あり
            -> 基本はその注文への更新依頼として扱う
            -> 商品名だけ / 数量だけ / 「追加で」も差分として解釈する

  -> 3. 何の連絡か分類する
       -> 現在注文なし
            -> 新規注文
            -> 在庫問い合わせ
            -> 品質不良 / 数量違い / 誤納品
            -> 請求確認
            -> 判別不能
       -> 現在注文あり
            -> 現在注文への追加
            -> 現在注文の数量変更 / 商品変更
            -> 現在注文の一部キャンセル / 全体キャンセル
            -> 納品条件変更
            -> 在庫問い合わせ
            -> 品質不良 / 数量違い / 誤納品
            -> 請求確認
            -> 判別不能

  -> 4. 変更対象を構造化する
       -> 商品名
       -> 数量
       -> 単位
       -> 追加 / 変更 / 取消の種別
       -> 納品日
       -> 納品時間帯
       -> 備考

  -> 5. 現在注文が編集可能か判定する
       -> 編集可
       -> 締切超過 / 出荷確定 / 配送中 -> 担当者確認

  -> 6. 在庫 / 納品可否を再判定する
       -> 全量OK
       -> 一部不足
       -> 全量不足
       -> 代替候補あり
       -> 分納可能
       -> 配送条件NG

  -> 7. 顧客へ返信する
       -> 内容更新を確定
       -> 確認質問
       -> 代替提案
       -> 分納提案
       -> 担当者確認中

  -> 8. 状態保存
       -> 受注済み
       -> 要対応
       -> awaiting_reply
       -> キャンセル
```

### 3.2. 現在注文ありの場合の分岐

```text
現在注文あり
  -> 追加依頼
       -> 商品追加 + 在庫再確認

  -> 数量変更 / 商品変更
       -> 対象商品が一意に分かる -> 差分反映
       -> 対象商品が曖昧 -> 確認質問

  -> 一部キャンセル
       -> 該当商品だけ削除 / 数量減

  -> 全体キャンセル
       -> 編集可なら取消
       -> ロック済みなら担当者確認

  -> 納品条件変更
       -> 再計算して反映可能
       -> 不可なら代替時間 / 担当者確認

  -> 納品後トラブル
       -> 品質不良
       -> 数量違い
       -> 誤納品
       -> 請求差異
```

## 4. 主要分岐一覧

| ID | 分岐 | 典型トリガー | システム判断 | 顧客への返し方 | 保存状態 |
|---|---|---|---|---|---|
| B-01 | 新規注文開始 | 現在注文なし + 商品・数量あり | 新規 draft 作成、在庫確認 | 内容を復唱して受注確定 or 確認質問 | `受注済み` / `awaiting_reply` |
| B-02 | 現在注文への追加 | 「白菜も追加で」「牛乳2ケースお願いします」 | 現在注文に商品追加、在庫再確認 | 追加後の現在注文を復唱 | `受注済み` / `awaiting_reply` |
| B-03 | 数量変更 / 商品変更 | 「トマトを15kgに」「卵を2箱に」 | 対象行を更新、在庫再確認 | 更新後の内容を復唱 | `受注済み` / `awaiting_reply` |
| B-04 | 一部キャンセル | 「卵だけなしで」「しめじは外してください」 | 対象行を削除 or 数量減 | 残る注文内容を復唱 | `受注済み` |
| B-05 | 全体キャンセル | 「やっぱり全部キャンセルで」 | 編集可なら取消、ロック済みなら要確認 | 取消完了 or 担当者確認中 | `キャンセル` / `要対応` |
| B-06 | 納品条件変更 | 「午後便にして」「明日納品に変更」 | 締切・配送条件を再計算 | 変更確定 or 代替提案 | `受注済み` / `awaiting_reply` / `要対応` |
| B-07 | 変更対象あいまい | 「さっきのやつを減らして」 | 対象商品や数量が特定できない | 何をどう変えるか確認 | `awaiting_reply` |
| B-08 | 在庫切れ | 全品在庫0 | お断りメッセージ（テンプレート）、需要把握用にNEEDS_REVIEWで保存 | 「在庫が0のため受け付けられません。ご了承ください。」 | `要対応` |
| B-09 | 部分在庫 | 一部しか確保できない | 確保可能数量を提示して顧客に確認（テンプレート） | 「○○{available}でよろしいですか？」 | `awaiting_reply` |
| B-10 | 締切超過 / ロック済み | 出荷確定後の変更依頼 | 自動反映せず人手確認 | 「担当者が確認します」 | `要対応` |
| B-11 | 現在注文なしで変更依頼 | 「さっきの卵をなしで」だが注文がない | 対象注文なし | 新規注文か確認、または担当者確認 | `awaiting_reply` / `要対応` |
| B-12 | 品質不良 / 数量違い / 誤納品 | 「2箱しか来ていない」「傷みが多い」 | 新規受注ではなくトラブル受付 | 謝意 + 確認 + 次アクション | `要対応` |
| B-13 | 請求確認 | 単価・明細差異 | 請求書 / 受注照合 | 担当確認中と案内 | `要対応` |
| B-14 | 既読のみ・無応答 | 重要確認に返答なし | SLA 超過 | 電話 / 担当者連絡へ切替 | `要対応` |

## 5. LINE 会話テンプレート集

### 5.1. 新規注文

**飲食店 → 卸**
> 明日納品で、鶏もも2kg、玉ねぎ10kg、卵3箱お願いします

**卸 → 飲食店**
> いつもお世話になっております。  
> ご注文を承りました。  
> - 鶏もも 2kg  
> - 玉ねぎ 10kg  
> - 卵 3箱  
> 納品予定は 5/26 午前便です。

### 5.2. 現在注文への追加

**飲食店 → 卸**
> 白菜も追加でお願いします

**卸 → 飲食店**
> 承知いたしました。  
> 現在のご注文に **白菜** を追加いたします。  
> 更新後の内容は以下です。  
> - 鶏もも 2kg  
> - 玉ねぎ 10kg  
> - 卵 3箱  
> - 白菜 2ケース  
> この内容で手配いたします。

### 5.3. 数量変更

**飲食店 → 卸**
> さっきのトマト、15kgに変更してください

**卸 → 飲食店**
> 承知いたしました。  
> 現在のご注文の **トマトを 15kg** に変更いたします。  
> 他の商品は先ほどの内容のままで手配いたします。

### 5.4. 一部キャンセル

**飲食店 → 卸**
> 卵だけなしでお願いします

**卸 → 飲食店**
> 承知いたしました。  
> 現在のご注文から **卵 3箱** を取り消しいたします。  
> そのほかの商品はこのまま手配いたします。

### 5.5. 全体キャンセル

**飲食店 → 卸**
> やっぱり全部キャンセルでお願いします

**卸 → 飲食店**
> 承知いたしました。  
> 現在のご注文は取り消しいたします。  
> すでに出荷準備後の場合は、別途ご連絡することがあります。

### 5.6. 納品条件変更

**飲食店 → 卸**
> 今日の便、午後でお願いします

**卸 → 飲食店**
> 承知いたしました。  
> 現在のご注文を **本日午後便** に変更できるか確認いたします。  
> 調整可能な場合は、その内容で手配いたします。

### 5.7. 在庫切れ（全品在庫0）

**飲食店 → 卸**
> しめじも追加でお願いします

**卸 → 飲食店**（テンプレート返答、LLMを通さない）
> しめじは在庫が0パックのため1パックの注文を受け付けられません。  
> ご了承ください。

※ 注文はNEEDS_REVIEWステータスでDBに保存（需要把握・機会損失分析用）。担当者への転送は行わない。

### 5.8. 部分在庫（一部のみ確保可能）

**飲食店 → 卸**
> 鶏むねを10kg追加で

**卸 → 飲食店**（テンプレート返答、LLMを通さない）
> 鶏むねは在庫が6kgです。10kgには4kg不足しています。  
> 鶏むね 6kgでよろしいですか？

※ 顧客がOK→部分数量で注文確定。顧客がNG→注文せず終了。勝手に部分数量で注文確定しない。

### 5.9. 変更対象が不明

**飲食店 → 卸**
> さっきのやつ減らしてください

**卸 → 飲食店**
> 確認です。  
> 現在のご注文のうち、どの商品を変更いたしましょうか。  
> 商品名と数量をお知らせください。

### 5.10. 現在注文が見つからない

**飲食店 → 卸**
> さっきの卵なしで

**卸 → 飲食店**
> 確認です。  
> 変更対象の現在注文が見当たりませんでした。  
> 新しいご注文として承る場合は、商品名と数量をあらためてお知らせください。

### 5.11. ロック済みで自動変更不可

**飲食店 → 卸**
> もう出てる便だけど、卵を追加したい

**卸 → 飲食店**
> 申し訳ありません。現在のご注文は出荷手配が進んでいるため、LINE 上での自動変更ができません。  
> 担当者が確認のうえ、ご連絡いたします。

### 5.12. 品質不良 / 数量違い

**飲食店 → 卸**
> 卵3箱のはずが2箱でした

**卸 → 飲食店**
> ご連絡ありがとうございます。確認いたします。  
> 不足分の追加手配または請求修正について、担当者よりご案内いたします。

## 6. システム要件整理

### 6.1. 意図分類要件

LINE メッセージ受信時に、少なくとも以下を判別できる必要がある。

1. **新規注文**
2. **現在注文への追加**
3. **現在注文の数量変更 / 商品変更**
4. **現在注文の一部キャンセル**
5. **現在注文の全体キャンセル**
6. **納品時間・納品日変更**
7. **在庫問い合わせ**
8. **品質不良 / 数量違い / 誤納品**
9. **請求確認**
10. **肯定返信 / 否定返信 / 保留返信**
11. **判別不能**

特に LINE では、以下の短文を個別に扱う。

- 肯定: `OK`, `それで`, `お願いします`, `はい`
- 否定: `やめます`, `不要です`, `今回はなしで`
- 差分指示: `追加で`, `やっぱり`, `さっきの`, `それは明日で`

### 6.2. 現在注文の解決要件

まず、顧客ごとに **現在注文** を特定する。

| 判定 | 内容 |
|---|---|
| `current_order` あり | その注文への更新依頼として処理する |
| `current_order` なし | 新規注文として処理する |
| `current_order` はあるがロック済み | 自動更新せず `要対応` に回す |

**ルール**

1. 顧客に編集可能な現在注文がある間は、商品名だけのメッセージも **追加依頼** とみなす
2. 顧客に編集可能な現在注文がある間は、`追加で` `やっぱり` `それは明日で` などの差分表現を **同一注文の更新** とみなす
3. `追加` / `変更` / `キャンセル` の語があっても、現在注文がなければ **対象注文なし** として確認する
4. 顧客が明示的に別注文を作る運用は MVP では想定しない
5. もし業務上まれに「現在注文を残したまま別注文」が必要になった場合は、自動新規作成せず `要対応` に送る
6. 受注Noは内部管理にのみ使い、LINE 返信には含めない

### 6.3. 業務判定要件

現在注文の作成または更新の前に、少なくとも以下を判定する。

| 判定 | 必須内容 |
|---|---|
| 顧客特定 | `tenant_id`, `customer_id`, `channel_user_id` |
| 現在注文特定 | `current_order_id`, `status`, `editable` |
| 商品正規化 | `product_id`, 表記ゆれ候補, 代替候補 |
| 数量妥当性 | `quantity`, `unit`, 不自然な数量, 不定貫判定 |
| 納品条件 | `delivery_date`, `delivery_slot`, リードタイム, 締切 |
| 在庫可否 | 全量可 / 一部可 / 全量不可 |
| 対応方針 | 即確定 / 顧客確認 / 担当者確認 |

**ルール**

1. 情報が欠けている更新依頼は **自動反映しない**
2. 在庫不足の更新は **受注済みのまま自動確定しない**
3. 締切超過や出荷確定後の変更は **自動反映しない**
4. 品質不良や請求確認は **新規受注として保存しない**

### 6.4. 保存データ要件

最低限、以下の情報を保持する。

| 項目 | 用途 |
|---|---|
| `message_history` | LINE 会話継続、短文返信解釈 |
| `current_order_id` | 顧客の現在注文 |
| `current_order_snapshot` | 顧客向けに復唱すべき最新注文内容 |
| `current_order_editable` | 自動更新可能かどうか |
| `pending_order_draft` | 新規注文または変更確認待ちの仮内容 |
| `pending_action_type` | 追加 / 変更 / 一部キャンセル / 全体キャンセル / 条件変更 |
| `confirmation_context` | 何を確認中か（代替、数量、納品日など） |
| `stock_check_result` | 在庫OK / 一部不足 / 代替候補 / 分納候補 |
| `cutoff_result` | 締切内 / 締切後 |
| `order_revision_log` | 追加・変更・取消の履歴 |
| `escalation_reason` | 人手対応に送った理由 |

Intent は `src/services/intent_understanding.py` で、まず `new_order` / `modify_current_order` / `partial_cancel` / `full_cancel` / `repeat_previous_order` / `repeat_usual_order` / `inventory_inquiry` / `order_status_inquiry` / `unclear` に分類する。
明確な全体キャンセル文言は即時ルールで分類し、現在注文がある状態で「やめとこうかな」のようにルールだけでは曖昧な文言は LLM に JSON Intent 分類させる。
LINE セッションの `pending_action_type` は、この分類済み Intent から導出する。`repeat_usual_order` / `repeat_previous_order` は `OrderMemoryService` で注文ドラフトへ復元し、在庫確認後に通常の受注確定フローへ流す。

### 6.5. 状態管理要件

| レイヤー | 状態 | 意味 |
|---|---|---|
| セッション | `active` | 会話継続中 |
| セッション | `awaiting_reply` | 顧客の返答待ち |
| セッション | `completed` | 会話が完了 |
| 受注 | `受注済み` | 現在注文として継続中。変更可能な場合あり |
| 受注 | `要対応` | 人手確認が必要 |
| 受注 | `配送中` | 原則ロック済み |
| 受注 | `キャンセル` | 取消確定 |
| 受注 | `完了` | 納品完了。次回は新規注文として扱う |

### 6.6. 返信テンプレート管理要件

LINE の文面は LLM に全文自由生成させず、**テンプレート優先** とする。

推奨構成:

```text
_templates/
  line/
    line_settings.json
    order_confirm.txt
    order_add_confirm.txt
    order_change_confirm.txt
    order_partial_cancel_confirm.txt
    order_full_cancel_confirm.txt
    order_change_clarify.txt
    order_no_current_order.txt
    stock_alternative.txt
    stock_partial_fulfillment.txt
    order_locked_need_review.txt
    trouble_ack.txt
```

LLM の役割は、主に以下に絞る。

1. 意図分類
2. 差分内容の抽出
3. 対象商品の特定補助

文面そのものは、`reply_type + variables` でテンプレ描画する。

### 6.7. 返信ポリシー要件

1. **未確定時は確定表現を使わない**
   - NG 例: `承りました`, `確定しました`
   - OK 例: `確認いたします`, `ご希望をお知らせください`
2. **確定時は更新後の注文内容を復唱する**
3. **代替提案は選択肢を明示する**
4. **締切後やロック済みでは、できること / できないことを明示する**
5. **トラブル系は謝意・確認・次アクションをセットで返す**
6. **LINE 返信には受注Noを書かない**

### 6.8. ダッシュボード / オペレーション要件

担当者画面には、以下の切り口が必要になる。

1. **要対応キュー**
   - 在庫不足
   - 代替返答待ち
   - 対象商品不明
   - ロック済み変更依頼
   - 品質不良
   - 請求差異
2. **現在注文の最新スナップショット表示**
3. **会話履歴表示**
4. **pending draft / pending action 表示**
5. **担当者が手動で変更確定 / 取消 / 返信できる操作**
6. **変更履歴 / 取消履歴**

### 6.9. テスト観点

最低限、以下はシナリオテストに入れる。

1. 現在注文がない場合は新規注文として確定される
2. 現在注文がある場合、商品追加がその注文に差分反映される
3. 数量変更が対象商品に正しく適用される
4. 一部キャンセル後、残り注文内容が復唱される
5. 現在注文がないのに「キャンセル」が来た場合、確認質問へ分岐する
6. 在庫不足時に `受注済み` のまま危険な確定文を出さない
7. ロック済み注文への変更依頼が `要対応` に回る
8. `OK` だけの返信で pending draft が確定される
9. 品質不良メッセージが新規受注化されない
10. LINE 返信に受注Noが含まれない

## 7. MVP で優先すべき分岐

ハッカソンデモで優先度が高いのは、以下である。

1. **現在注文がない場合の新規受注**
2. **現在注文がある場合、新規注文を増やさず更新依頼として処理すること**
3. **現在注文がある場合の追加**
4. **現在注文がある場合の数量変更 / 一部キャンセル**
5. **単位・数量あいまい時の確認質問**
6. **在庫不足時の代替提案**
7. **ロック済み注文の `要対応` 振り分け**
8. **短文返信 (`OK`, `それで`) による確定**
