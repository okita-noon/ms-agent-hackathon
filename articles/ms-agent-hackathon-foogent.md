## 「りんご10箱とバナナ20kg、明日お願いね」――この一言から始まる、現場の苦労

食品卸や食材メーカーの受発注業務を覗いたことはありますか？

飲食店のオーナーが朝の仕込み前にLINEで「いつものお願い」と送ってくる。昼過ぎには電話で「トマト……えっと、いつもの量で」と言われる。夕方にはFAXで手書きの注文書が届く。

これらの注文を受ける側は、**1件1件を手作業でシステムに入力**しています。商品名の表記ゆれを読み解き（「ツナ缶」と「シーチキン」は同じ商品？）、数量の妥当性を経験で判断し（「トマト150kgって、いつも15kgなのに大丈夫？」）、在庫を確認して、配送ルートを組む。

中小の食品卸では、この受注処理だけで**1日2〜3時間**を費やすことも珍しくありません。

ここで、いくつかの数字を紹介させてください：

> - 中堅・中小企業の**85.8%が、FAX・電話などアナログ手段で受注**している[^1]
> - 食料・飲料卸売業の販売額は**年間約18.9兆円**[^2]——にもかかわらず、その受発注の大部分が未だに人手で処理されている
> - 日本の事業系食品ロスは**年間約231万トン**[^3]——その一因に、聞き間違い・転記ミスによる誤発注がある

つまり、**約19兆円市場の入口が、FAXと電話と手入力に支えられている**。この構造的なギャップにこそ、AIエージェントが介入する余地があると私たちは考えました。

[^1]: 株式会社アイル「[中堅・中小企業における受注業務の実態調査](https://prtimes.jp/main/html/rd/p/000000039.000027651.html)」（2020年、受注業務担当者734名対象）
[^2]: 経済産業省「[商業動態統計](https://www.meti.go.jp/statistics/tyo/syoudou/index.html)」2024年暦年・食料飲料卸売業（大規模卸店）販売額
[^3]: 農林水産省「[事業系食品ロス量（2023年推計値）](https://www.maff.go.jp/j/press/shokuhin/recycle/250627.html)」（2025年6月公表）

私たちチーム「ASKNOI_AI木曜会」は、この課題を**複数のAIエージェントが協調して解決するシステム**で挑みました。ターゲットは**従業員10〜50名規模の食品卸・食材メーカー**。大手向けEDI（電子データ交換）には手が届かないけれど、日々の電話・LINE受注に悲鳴を上げている、そんな現場です。

## つくったもの：「foogent」― AI受発注自動一元管理システム

**foogent**（フージェント）は、食品卸・食材メーカー向けの**マルチテナント対応 AI Agent SaaS**です。

LINEや電話から届く注文を、複数の専門AIエージェントがリレー形式で自動処理します。

**デモシナリオを4つ紹介します。それぞれが解決する業務課題とセットでご覧ください：**

**① 通常注文 → 受注処理時間の削減**
顧客がLINEで「りんご10箱、バナナ20kg」と送信 → AIが顧客を特定し、商品を正規化し、在庫を確認し、受注を確定 → ダッシュボードに即反映。**人間の介在ゼロ。** 従来5〜10分かかっていた1件の受注処理が、数秒で完了します。

**② 曖昧表現の学習 → 属人化の解消**
初回に「ツナ缶100g」と注文 → AIが「ツナ缶にはライトツナ缶とツナフレーク缶があります。どちらですか？」と確認 → 次回以降は自動で解釈。**使うほど賢くなる。** ベテラン担当者の頭の中にしかなかった「この顧客のツナ缶＝ライトツナ缶」という知識が、AIのデータとして永続化されます。

**③ 誤発注検知 → 食品ロスと損失の防止**
いつも15kgのトマトを注文する顧客が「トマト150kg」と送信 → AIが過去の発注パターンからZ-scoreを計算し、異常を検知 → 「通常15kgのところ150kgですが、お間違いないですか？」と確認。**10倍の誤発注を未然に防ぐ。** 食品は一度出荷すると返品が利かないケースが多く、この1回の検知が数万円の損失を防ぎます。

**④ いつもの注文 → 24時間対応と顧客満足度の向上**
常連客が「いつものお願い」と一言送るだけ → 学習済みの発注パターン（信頼度スコア0.9以上）から自動で注文を確定。**たった5文字で受注完了。** 深夜や早朝に送られた注文も即座に処理され、営業時間外の機会損失がなくなります。

## なぜ作ったか ― 食品業界の「DXの谷」と、その経済的損失

食品業界のIT化率は他業界と比べて低く、特に中小の食品卸では受発注業務が未だにアナログなままです。

経済産業省 中小企業庁の「受発注のデジタル化に関する推進方策 報告書」[^4]では、中小企業間の取引がFAX・電話中心であり、デジタル化が喫緊の課題であることが指摘されています。大手向けのEDIシステムは存在しますが、導入コストと運用コストが障壁となり、**中小企業にはハードルが高い**。また、食品製造業ではIoT・デジタル技術の活用率が**36.8%**にとどまり[^5]、人材不足と費用対効果への懸念がデジタル化を阻んでいます。結果として、電話・FAX・LINEという「人間が受けるチャネル」が主流のまま残っています。

### 「アナログ受注」が引き起こす3つの経済的損失

私たちが着目したのは、この構造的な課題が引き起こす**具体的なコスト**です：

**① 人件費の浪費：受注処理に年間500〜700時間**
中小食品卸の受注担当者が1日2〜3時間を受注処理に費やすとすると、年間約500〜700時間。なお、アイルの同調査[^1]では、受注業務をデジタル化した企業の約8割が業務時間の削減を実感しており、1人あたり1日5時間以上の削減に成功した企業もあります。しかもこの業務は、ドライバー不足や人件費高騰が深刻化する食品卸にとって、最も自動化すべき領域です。

**② 誤発注による食品ロスと信頼損失**
電話の聞き間違い、FAXの読み間違い、手入力のタイプミス。食品業界の受注トラブルとして、「電話対応での聞き取りミス」「FAXの文字が読めず誤納品」「担当者不在時に"いつもの"を処理できない」といった事例が報告されています[^6]。食品は返品しても再販できないケースが多く、1件の誤発注がそのまま食品ロスと損失につながります。

**③ 属人化リスク：担当者が辞めたら業務が止まる**
「Aさんの"いつもの"はりんご10箱とバナナ20kg」「Bさんが"ツナ缶"と言ったらライトツナ缶のこと」——こうした暗黙知が**特定の担当者の頭の中にしかない**。担当者が休んだり退職したりすると、受注品質が一気に低下します。食品製造業の労働生産性は製造業全体の約60%の水準にとどまっており[^7]、こうした属人的業務の効率化は業界全体の課題です。

[^4]: 経済産業省 中小企業庁「[受発注のデジタル化に関する推進方策 報告書](https://www.chusho.meti.go.jp/keiei/gijut/digitalization/download/report_r3.pdf)」（2022年3月）
[^5]: 富士電機「[食品製造業におけるIoT/ITの利用動向調査](https://www.fujielectric.co.jp/products/foodfactory/solution_detail/research_research02.html)」
[^6]: フーズチャネル「[食品卸売業の受注トラブル事例まとめ](https://foods-ch.infomart.co.jp/knowledge/seller/1646272989832)」
[^7]: 農林水産省「[食品製造業等の生産性向上](https://www.maff.go.jp/j/shokusan/sanki/soumu/seisansei.html)」

これらの課題を整理すると、以下の4つに集約されます：

- **チャネルが分散**している（LINE・電話・FAX・メール）
- **表記ゆれ**が日常的に発生する（「りんご」「リンゴ」「林檎」「ふじ」）
- **暗黙知**に依存している（「いつもの」「多めで」の解釈は担当者の記憶頼み）
- **ミスの発見が遅れる**（誤発注に気づくのは配送後ということも）

これらはまさに、**AIエージェントが得意とする領域**です。自然言語の理解、パターン学習、異常検知、マルチチャネル対応——単一のAIではなく、**専門性を持った複数のエージェントが協調する**ことで、はじめて現場の複雑さに対応できると考えました。

### なぜルールベースやRPAではダメなのか

「それ、if文とRPAで十分じゃない？」と思われるかもしれません。でも、現場の注文はルール化できないものだらけです。

「りんご」「リンゴ」「林檎」「ふじりんご」——同じ商品に対する呼び方が顧客ごとに違う。「いつもの」の中身は顧客と季節で変わる。「多めで」の具体的な量は文脈依存。これらを事前にすべてルール化するのは非現実的です。

AIエージェントなら、自然言語を柔軟に解釈し、過去の注文パターンから学習し、**ルールを自ら作り出す**ことができます。しかも各エージェントが専門領域に集中するマルチエージェント構成にすることで、単一LLMに全責務を負わせるより高い精度と説明可能性を実現しました。

## AIエージェントの設計 ― 「専門家チーム」というアプローチ

foogentの核心は、**4つの専門エージェント＋1つのオーケストレーター**による協調処理です。

### なぜマルチエージェントなのか

受発注処理は一見シンプルに見えますが、実際には複数の専門知識が必要です。顧客の特定、商品名の正規化、異常検知、在庫管理、メッセージ生成——これらを1つのプロンプトに詰め込むと、責務が曖昧になり、精度が落ちます。

私たちは「**人間の業務チームと同じ構造**」をAIで再現しました：

### Intake Agent（受付担当）

顧客特定・商品名の正規化・過去パターンの照合を担当します。

```python
@kernel_function(
    name="normalize_product",
    description="あいまいな商品名を正式名称にマッチングする",
)
async def normalize_product(self, raw_name, tenant_id):
    master = self._ctx.get_connector("IProductMaster")
    candidates = await master.fuzzy_match(tenant_id, raw_name, limit=5)
    return [c.model_dump() for c in candidates]
```

「ツナ缶」「シーチキン」「まぐろ油漬」——顧客が使う多様な表現を、SQL LIKE検索やAI Searchのベクトル検索を組み合わせて正式な商品名に解決します。

### Exception Agent（検品担当）

注文内容の異常を検知します。**Z-scoreベースの統計的異常検知**が特徴です。

```python
# 過去の発注パターンからZ-scoreを計算
z_score = abs(ordered_qty - stats.avg_qty) / stats.std_dev
threshold = profile.anomaly_thresholds.get("qty_z_score", 3.0)
is_anomaly = z_score > threshold
```

「いつも15kgなのに150kg」のような桁間違いを、顧客ごとの発注履歴から統計的に検出します。単位の異常（いつもは「箱」なのに「kg」で注文）も検知対象です。

### Inventory Agent（在庫担当）

在庫の確認・代替品の提案・在庫引当を行います。

在庫不足の場合は同カテゴリの代替品を自動検索し、「有機トマトは在庫切れですが、ミニトマトなら30kg在庫があります」といった提案を生成します。

### Communication Agent（応対担当）

顧客へのLINEメッセージを生成します。確認質問、警告、在庫不足の通知など、状況に応じた自然な日本語メッセージを作成します。

### Orchestrator ― エージェント間の司令塔

Semantic Kernelの `ChatCompletionAgent` を使い、上記4エージェントをパイプラインで呼び出します。

```
ユーザーメッセージ
  → Intake Agent（顧客特定・商品解決）
    → Exception Agent（異常検知）
      → Inventory Agent（在庫確認・引当）
        → 受注保存
          → Communication Agent（返信生成）
            → LINE返信
```

オーケストレーターは**意図分類**も行います。「新規注文」「確認への返答」「いつもの注文」「質問」を判別し、適切なフローに振り分けます。

## プロンプト設計の工夫

AIエージェントの品質はプロンプトで決まります。私たちが特に工夫した点を紹介します。

### 1. 「確認しすぎない」ための信頼度スコア

すべてを確認質問していたら、人間より遅くなります。そこで**信頼度スコア**による3段階の制御を導入しました：

- **0.9以上**：自動確定（過去に同じ注文パターンが複数回あり、完全一致）
- **0.5〜0.9**：軽い確認（「りんご10箱でよろしいですか？」）
- **0.5未満**：必ず確認（商品が特定できない、数量が曖昧など）

これにより、常連客の「いつものお願い」は確認なしで即座に処理され、初めての注文や曖昧な表現のときだけ丁寧に確認する、**人間の熟練スタッフと同じような判断**が可能になりました。

### 2. 日本語特有の表現への対応

食品業界では独特の表現が飛び交います。プロンプトには以下のような業務知識を埋め込みました：

- **不定貫**（ふていかん）：果物や肉など、1個あたりの重量が一定でない商品。注文時は概算、ピッキング時に実測。
- **温度帯**：常温・冷蔵・冷凍の3区分。配送ルートとキャリアの選択に直結。
- **単位変換**：「1箱」「10kg」「5パック」——同じ商品でも顧客によって使う単位が違う。

### 3. 会話履歴の活用

LINEでの注文は1回のメッセージで完結しないことがあります。「トマト10箱」→「あ、やっぱり15箱で」→「あとバナナも追加して」のような**マルチターン会話**に対応するため、セッション管理とメッセージ履歴の仕組みを実装しました。

Cosmos DBにセッション情報をTTL付き（2時間）で保存し、会話の文脈を維持します。

## Connector層 ― マルチテナントを支える設計

foogentは**SaaS**として設計されており、複数の食品卸企業（テナント）が同一システムを利用できます。

この実現のために採用したのが**Connector パターン**です。

```python
class TenantContext:
    def get_connector(self, interface_name: str) -> Any:
        return self._factory.resolve(interface_name)
```

すべてのAgent・Pluginは `TenantContext.get_connector("インターフェース名")` 経由でデータにアクセスします。**DBやAPIへの直接アクセスは一切禁止。** これにより、テナントごとに異なるDB接続先やAPI設定を、ビジネスロジックを一切変えることなく差し替えられます。

6つのインターフェース（`IOrderRepository`、`IProductMaster`、`IInventoryService`、`ICustomerRepository`、`ISessionRepository`、`IOrderIntelligenceStore`）を Protocol で定義し、Cosmos DBアダプタやAzure SQLアダプタとして実装しています。

新しいテナントの追加は、ConnectorFactoryにテナント設定を追加するだけ。コードの変更は不要です。

## 学習する受注AI ― Learning Service

foogentの最大の特徴は**使うほど賢くなる**ことです。

注文が確定するたびに、バックグラウンドで `LearningService` が動きます：

1. **パターン記録**：顧客×商品×数量×単位の組み合わせをCosmos DBに保存
2. **プロファイル更新**：顧客ごとの発注傾向（平均数量・標準偏差・よく使う単位）を統計的に更新

この蓄積が、次回以降の注文で威力を発揮します：

- 「いつもの」の自動解釈（信頼度スコアの向上）
- 異常検知の精度向上（統計量の充実）
- 表記ゆれの学習（顧客固有の商品呼称の記録）

## インフラとCI/CD

### Azure上の構成

Bicepテンプレート（13モジュール）でインフラをコード管理しています。主なリソース：

- **Azure Container Apps**：FastAPIアプリのホスティング（min=0でスケールイン、コスト最適化）
- **Azure Cosmos DB Serverless**：受注ドキュメント・セッション・学習データ（サーバーレスで従量課金）
- **Azure SQL Database Basic**：商品マスタ・顧客・在庫（リレーショナルデータ）
- **Azure Key Vault**：接続文字列・APIキーの安全な管理

### GitHub Actions による自動デプロイ

`main` ブランチへのpushをトリガーに、以下のパイプラインが自動実行されます：

```
コード push → ACR Build → Push（SHA tag + latest）
  → Container Apps Update → Health Check
```

## 苦労した点・学び

### LINEのreply tokenは30秒で失効する

LINE Messaging APIの `replyToken` は30秒の有効期限があります。マルチエージェントの処理が30秒を超えると返信できなくなるため、タイムアウト時は `push` メッセージにフォールバックする仕組みを実装しました。

### Webhook の重複イベント対策

LINEのWebhookは同じイベントを複数回送信することがあります。インメモリのTTLキャッシュ（`_EventDedup`）で重複排除を実装し、同じ注文が2回処理されることを防いでいます。

### セキュリティへの配慮

食品卸の受注データには顧客情報が含まれるため、セキュリティには特に注力しました：

- LINE Webhookの **HMAC-SHA256署名検証**
- Azure Key Vaultによる秘密情報の一元管理
- pre-commitフックによる `.env` ファイルのコミット防止
- Entra IDによる認証

### 「AIは0から1ではなく、人間の1を90に引き上げる存在」

開発を通じて実感したのは、AIエージェントは人間の仕事を「なくす」のではなく、**人間が持っている業務知識をスケールさせる**ものだということです。

熟練スタッフが持つ「この顧客はいつもこの量を注文する」「この商品名はあの商品のことだ」という暗黙知を、AIがデータとして学習し、24時間稼働で再現する。人間はより創造的な仕事——新規顧客の開拓やメニュー提案——に集中できるようになります。

## foogentが生み出すビジネスインパクト

ここで、foogentが実際にどれほどの価値を生むか、具体的に試算してみます。

### 導入効果の試算（従業員30名規模の食品卸の場合）

| 項目 | Before（手作業） | After（foogent） | 削減効果 |
|------|-----------------|-----------------|---------|
| 受注処理時間 | 2.5時間/日（受注担当1名） | 0.5時間/日（例外対応のみ） | **年間約500時間削減** |
| 誤発注率 | 月2〜3件（聞き間違い・転記ミス） | 異常検知で月0〜1件に | **誤発注コスト年間数十万円削減** |
| 属人化リスク | 担当者依存（退職時に業務品質低下） | AIが学習データとして蓄積 | **業務知識の永続化** |
| 夜間・早朝対応 | 対応不可（営業時間外） | 24時間自動受注 | **機会損失ゼロ** |
| 新人教育コスト | 3〜6ヶ月のOJT | 初日からAIが補佐 | **立ち上がり期間を大幅短縮** |

アイルの調査[^1]では、受注業務をデジタル化した企業の**約8割が業務時間を削減**できており、中には1人あたり1日5時間以上の削減に成功した例もあります。foogentはAIエージェントによる自動処理のため、従来のWeb受発注システム以上の削減効果が期待できます。

### SaaS展開による市場インパクト

foogentはマルチテナント設計のSaaSです。経済産業省の商業動態統計[^2]によれば食料・飲料卸売業の販売額は年間約18.9兆円。この巨大市場を支える中小卸売企業の多くがアナログ受注に依存しています。

foogentのConnectorパターンにより、テナントごとの設定追加だけで新規企業を受け入れられるため、スケーラブルに展開可能です。さらに重要なのは、学習機能が蓄積するデータの価値です。顧客ごとの発注パターン、季節変動、商品の人気度——これらのデータは在庫最適化や需要予測にも活用でき、**食品サプライチェーン全体の効率化**につながるポテンシャルを秘めています。

### 食品ロス削減への貢献

誤発注の防止は、直接的に食品ロスの削減に貢献します。農林水産省の公表[^3]によれば、2023年度の事業系食品ロスは約231万トン。この中には、受発注プロセスで生じる過剰発注や誤発注に起因するものも含まれます。

foogentのException Agentによる統計的異常検知は、この「発注段階での食品ロス」を未然に防ぐ仕組みです。たとえば「いつも15kgのところ150kgの注文」を検知して確認を促すだけで、135kg分の不要な食品生産・廃棄を防げます。SDGs目標12「つくる責任 つかう責任」の達成に、テクノロジーの側面から貢献できるプロダクトです。

## まとめ

foogentは、食品卸の受発注業務という**19兆円市場を支える基幹業務のアナログ課題**に対して、**複数のAIエージェントが協調するアーキテクチャ**で挑んだプロダクトです。

- **社会課題**：85.8%がアナログ受注[^1]、事業系食品ロス年間231万トン[^3]。人手不足・属人化が深刻
- **ビジネスインパクト**：デジタル化企業の約8割が業務削減を実現[^1]。誤発注の激減、24時間受注による機会損失ゼロ
- **AIエージェントの活用**：4つの専門エージェントがパイプラインで協調し、受注→異常検知→在庫確認→返信を自動化
- **学習機能**：使うほど賢くなり、常連客の「いつもの」を理解し、属人化を解消する
- **スケーラビリティ**：マルチテナントSaaS＋Connectorパターンにより、食品卸以外の業種にも展開可能

## 今後の展望

MVP段階では実装できなかったものの、次のステップとして構想しているものがあります：

- **AI Searchによるベクトル検索**：現在はSQL LIKE検索ですが、AI Searchのベクトル検索を導入すれば「りんご」「リンゴ」「林檎」の表記ゆれに加え、意味的に近い商品の提案も可能に
- **電話チャネルの完全対応**：Azure Communication Services＋AI Speechによる音声認識パイプラインの構築（リソースはデプロイ済み）
- **ピッキングリストの自動生成**：温度帯・配送ルート別にピッキングリストをPDF出力し、倉庫業務まで一気通貫で効率化

## おわりに

Microsoft Agent Hackathon 2026 への参加は、チームにとって大きな学びの機会でした。限られた期間の中で、業務ドメインの深い理解とAIエージェント技術の両方が求められるという、まさにハッカソンならではの濃密な経験ができました。

特に印象的だったのは、**技術ドリブンではなく課題ドリブンで考えることの大切さ**です。最初は「マルチエージェントでかっこいいものを作りたい」という気持ちが先行していましたが、食品卸の業務を深く理解する中で、本当に必要な機能が見えてきました。不定貫、温度帯、配送ルート——現場を知らなければ絶対に設計できないドメイン知識の数々に、チーム全員が「業務理解あってこその技術だ」と痛感しました。

食品業界に限らず、「電話やメッセージで注文を受ける」業務は日本中に存在します。花屋、印刷会社、建材卸、部品商社——foogentのConnectorパターンとマルチエージェント構成は、業種を問わず応用できるアーキテクチャです。

**85%超のアナログ受注[^1]を変えるのは、大規模なシステム導入ではなく、「いつものLINE」にAIエージェントが寄り添うことかもしれません。** 顧客は今まで通りLINEで注文するだけ。裏側でAIが仕事をする。この「顧客体験を変えずにバックオフィスを変革する」アプローチこそが、中小企業のDXの現実解だと私たちは信じています。
