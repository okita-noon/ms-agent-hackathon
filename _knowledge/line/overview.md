# LINE受発注 業務ナレッジ — 全エージェント共通

> すべてのエージェントはこのドキュメントを最初に読むこと。

## 1. 業務の大前提

- 食品卸（売り手）が、飲食店（買い手）からの注文をLINEで受ける業務である
- 顧客は短文・省略・追記・修正が多い。LINEならではの崩した表現が頻出する
- 1顧客につき、配送完了まで開いている注文は **1件だけ**（1顧客1オープン注文）
- 配送完了前に届くLINEは、基本的に **現在注文の更新依頼** として扱う
- 配送完了後に届くLINEは **新規注文** として扱う
- LINEの返信には **受注Noを表示しない**（内部管理専用）

## 2. 現在注文の判断ルール

| 状況 | 扱い |
|---|---|
| 現在注文なし | 新規注文として受ける |
| 現在注文あり + 注文系メッセージ | その現在注文への更新依頼 |
| 現在注文あり + 編集不可（出荷確定後等） | 自動変更せず `要対応` に送る |

### 重要なルール

1. 顧客に編集可能な現在注文がある間は、商品名だけのメッセージも **追加依頼** とみなす
2. `追加で` `やっぱり` `それは明日で` などの差分表現は **同一注文の更新**
3. `追加` `変更` `キャンセル` の語があっても、現在注文がなければ **対象注文なし** として確認する
4. 同じ顧客の別注文を並行で作ることはMVPでは想定しない

## 3. 解釈の優先順位

会話履歴と現在注文は、以下の優先順で解釈する。

1. **pending_order_draft がある** → まず確認中の内容に対する肯定/否定/修正として解釈
2. **pending_order_draft はないが current_order がある** → 現在注文への追加/変更/キャンセル
3. **pending_order_draft も current_order もない** → 新規注文または問い合わせ

## 4. メッセージ分類（意図分類）

LINEメッセージは以下のいずれかに分類する。

| ID | 分類 | 例 |
|---|---|---|
| B-01 | 新規注文 | 「明日納品で鶏もも2kg、玉ねぎ10kgお願いします」 |
| B-02 | 現在注文への追加 | 「白菜も追加で」「牛乳2ケースお願いします」 |
| B-03 | 数量変更/商品変更 | 「トマトを15kgに」「卵を2箱に」 |
| B-04 | 一部キャンセル | 「卵だけなしで」「しめじは外してください」 |
| B-05 | 全体キャンセル | 「やっぱり全部キャンセルで」 |
| B-06 | 納品条件変更 | 「午後便にして」「明日納品に変更」 |
| B-07 | 変更対象あいまい | 「さっきのやつ減らして」 |
| B-08 | 在庫不足・代替あり | （在庫確認結果による） |
| B-09 | 在庫不足・分納可 | （在庫確認結果による） |
| B-10 | ロック済み変更依頼 | 「もう出てる便だけど卵追加したい」 |
| B-11 | 現在注文なしで変更依頼 | 「さっきの卵なしで」（注文がない） |
| B-12 | 品質不良/数量違い | 「卵3箱のはずが2箱でした」「傷みが多い」 |
| B-13 | 請求確認 | 単価・明細の差異 |
| B-14 | 既読のみ・無応答 | 重要確認に返答なし |

### LINEの短文パターン

- 肯定: `OK`, `それで`, `お願いします`, `はい`, `それでお願いします`
- 否定: `やめます`, `不要です`, `今回はなしで`
- 差分指示: `追加で`, `やっぱり`, `さっきの`, `それは明日で`

## 5. 状態の種類

| レイヤー | 状態 | 意味 |
|---|---|---|
| セッション | `active` | 会話継続中 |
| セッション | `awaiting_reply` | 顧客の返答待ち |
| セッション | `completed` | 会話が完了 |
| 受注 | `受注済み` | 現在注文として継続中 |
| 受注 | `要対応` | 人手確認が必要 |
| 受注 | `配送中` | ロック済み |
| 受注 | `キャンセル` | 取消確定 |
| 受注 | `完了` | 納品完了 |

## 6. 返信ポリシー

1. **未確定時は確定表現を使わない** — NG: `承りました` OK: `確認いたします`
2. **確定時は更新後の注文内容を復唱する**
3. **代替提案は選択肢を明示する**
4. **ロック済みでは、できること/できないことを明示する**
5. **トラブル系は謝意・確認・次アクションをセットで返す**
6. **LINE返信には受注Noを書かない**

## 7. エージェント構成

LINEの注文処理は以下の4エージェントが連携して行う。

| エージェント | 役割 | 詳細マニュアル |
|---|---|---|
| Intake Agent | 注文解析・顧客特定・商品正規化 | `intake_manual.md` |
| Exception Agent | 確認質問・異常検知 | `exception_manual.md` |
| Inventory Agent | 在庫照合・代替/分納判断 | `inventory_manual.md` |
| Communication Agent | 返信生成・テンプレート選択 | `communication_manual.md` |

処理順序: Intake → Exception（必要時） → Inventory → Communication
