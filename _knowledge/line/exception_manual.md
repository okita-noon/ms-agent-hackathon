# Exception Agent マニュアル — LINE

> 前提: `overview.md` を先に読んでいること。

## 1. このエージェントの責任

- 注文内容の異常検知（数量異常・単位異常・曖昧表現）
- 確認質問の生成（顧客に何をどう聞くか）
- 異常の深刻度判定（high / medium / low）

## 2. やらないこと

- 注文内容の構造化（Intake Agent の仕事）
- 在庫の確認（Inventory Agent の仕事）
- 返信メッセージの最終生成（Communication Agent の仕事）
- 異常を自分で修正する（確認を求めるだけ）

## 3. 判断ルール

### 3.1. 数量異常の検知

detect_quantity_anomaly で過去パターンとの乖離を確認する。

| 条件 | 判定 | severity |
|---|---|---|
| Zスコア > 3.0（通常の10倍以上） | 誤発注の可能性 | high |
| Zスコア > 2.0（通常の5倍程度） | 確認推奨 | medium |
| 過去データ3件未満 | 異常検知スキップ | — |

### 3.2. 単位異常の検知

detect_unit_anomaly で通常使用単位との不一致を確認する。

| 条件 | 判定 |
|---|---|
| 商品マスタの標準単位と異なる | 確認推奨 |
| 「kg」と「箱」の取り違えの可能性 | high |
| 個数と重量の混同 | medium |

### 3.3. 確認が必要なケース

| ケース | 確認質問の方向性 |
|---|---|
| 数量が通常の10倍以上 | 「〇〇を△△kgのご注文ですが、いつもは□□kg前後です。数量をご確認いただけますか？」 |
| 単位が通常と異なる | 「〇〇は通常△△(単位)でのご注文ですが、□□(単位)でよろしいですか？」 |
| 商品名が曖昧（複数候補あり） | 「〇〇は△△と□□がございます。どちらでしょうか？」 |
| 変更対象が特定できない | 「どの商品を変更いたしましょうか？」 |
| 数量の指定がない | 「〇〇は何kg（何箱）お送りしましょうか？」 |

### 3.4. 確認が不要なケース

| ケース | 理由 |
|---|---|
| 過去パターンと一致 | 正常な注文 |
| 数量・単位ともに明確 | 確認不要 |
| 肯定返信（OK、はい） | pending_draft の確定 |
| 過去データが3件未満 | 比較基準がないため検知スキップ |

## 4. 一問一答集

### カテゴリA: 数量異常の検知

### Q1. いつも10箱前後を注文する常連さんが突然100箱を注文してきた。これはOK？

**状況**: 過去6ヶ月の注文履歴が10件以上あり、りんごは平均10箱±2箱。今回100箱。

**注文内容の例**:
> Intake Agent から渡される注文ドラフト：
> ```json
> {
>   "customer_id": "C001",
>   "products": [{"name": "りんご", "quantity": 100, "unit": "箱"}],
>   "past_orders": [{"date": "2026-05-20", "quantity": 10}, {"date": "2026-05-13", "quantity": 9}, ...]
> }
> ```

**この顧客の過去パターン**: りんご平均10箱（min 8, max 14）

**正しい判断**: 
Zスコア > 3.0で「high」。100箱は通常の10倍。打ち間違い・桁間違いの可能性が高い。確認が必要。

**出力**:
```json
{
  "anomalies": [
    {
      "type": "quantity",
      "product": "りんご",
      "severity": "high",
      "message": "通常10箱前後のところ100箱のご注文です"
    }
  ],
  "confirmation_needed": true,
  "confirmation_message": "りんご100箱のご注文ですが、いつもは10箱前後です。数量をご確認いただけますか？"
}
```

**よくある間違い**:
- 「お客さんが言ってるんだから正しいはず」と通してしまう
- 勝手に10箱に修正してしまう

---

### Q2. いつも10箱の常連さんが12箱を注文してきた。これも確認する？

**状況**: 過去10件のりんご注文が平均10箱±2箱。今回12箱。

**正しい判断**: 
Zスコア < 2.0で正常範囲。12箱は通常の範囲内の増減なので確認不要。顧客体験を損なわないため、正常な変動幅として許容。

**出力**:
```json
{
  "anomalies": [],
  "confirmation_needed": false
}
```

**よくある間違い**:
- 少しでも普段と違うと全部確認してしまう（顧客体験が悪くなる）

---

### Q3. いつも10箱の常連さんが50箱を注文してきた。100箱ほどではないが多い。確認する？

**状況**: 過去パターン平均10箱±2箱。今回50箱。Zスコア > 2.0だが < 3.0。

**正しい判断**: 
Zスコア 2.0〜3.0は「medium」。5倍程度の増加は確認推奨。ただし high ほど緊急ではない。顧客がイベント仕込みなど意図的な大量発注の可能性もあるため、中立的に確認する。

**出力**:
```json
{
  "anomalies": [
    {
      "type": "quantity",
      "product": "りんご",
      "severity": "medium",
      "message": "通常10箱前後のところ50箱のご注文です"
    }
  ],
  "confirmation_needed": true,
  "confirmation_message": "りんご50箱のご注文ですが、いつもは10箱前後です。数量をご確認いただけますか？"
}
```

---

### Q4. 初めて注文してきたお客さんが大量に注文した。確認する？

**状況**: 過去データなし（初回注文）。鶏もも50kg。

**正しい判断**: 
過去データが3件未満なので異常検知をスキップ。比較対象がないので判断できない。初回顧客が最初から大量発注することもあり（業務用など）、データなしで異常扱いにはできない。

**出力**:
```json
{
  "anomalies": [],
  "confirmation_needed": false
}
```

**よくある間違い**:
- 「量が多いから」と過去データなしでも異常扱いにしてしまう
- 初回だから大量発注を拒否するような言い方をしてしまう

---

### Q5. 過去データが2件しかない顧客の注文で、2件とも5kgだったのに今回50kgが来た。確認する？

**状況**: 過去データ2件のみ（both 5kg）。基準不十分。

**正しい判断**: 
3件未満なので異常検知スキップ。統計的に有意な比較ができない。2件のデータでは「異常」か「通常の自然なばらつき」か判別できない。

**出力**:
```json
{
  "anomalies": [],
  "confirmation_needed": false
}
```

**よくある間違い**:
- 2件でも「明らかにおかしい」と判断してしまう

### カテゴリB: 単位異常の検知

### Q6. いつもkgで注文している商品を「箱」で注文してきた。確認する？

**状況**: トマトの過去注文は全て kg 単位（5kg, 8kg, 10kg等）。今回「トマト5箱」。

**注文内容の例**:
> Intake Agent から渡される注文ドラフト：
> ```json
> {
>   "customer_id": "C002",
>   "products": [
>     {"name": "トマト", "quantity": 5, "unit": "箱", "standard_unit": "kg"}
>   ]
> }
> ```

**正しい判断**: 
通常使用単位（kg）と異なる単位（箱）での注文。kgと箱では1注文あたりの量が大幅に変わる可能性があるため確認が必要。例: kg単位で5〜10kg程度、箱単位では1箱=3〜5kgが目安など、単位によって実量が大きく異なる。

**出力**:
```json
{
  "anomalies": [
    {
      "type": "unit",
      "product": "トマト",
      "severity": "medium",
      "message": "通常はkg単位でのご注文ですが、箱単位のご注文です"
    }
  ],
  "confirmation_needed": true,
  "confirmation_message": "トマトは通常kg単位でのご注文ですが、5箱でよろしいですか？"
}
```

**よくある間違い**:
- 「箱でもkgでも同じ商品だから問題ない」と通してしまう

**関連**: Q8

---

### Q7. 「個」と「kg」を間違えている可能性がある場合は？

**状況**: 玉ねぎを「100個」と注文。過去パターンでは「10kg」程度。

**正しい判断**: 
個数と重量の混同の可能性。severity = medium。単位が完全に異なる（個数 vs 重量）ため、顧客の意図を確認することが重要。個数を重さで発注する業務であれば、混同による大きなズレが生じる。

**出力**:
```json
{
  "anomalies": [
    {
      "type": "unit",
      "product": "玉ねぎ",
      "severity": "medium",
      "message": "通常はkg単位でのご注文ですが、個単位のご注文です"
    }
  ],
  "confirmation_needed": true,
  "confirmation_message": "玉ねぎは通常kg単位でのご注文ですが、100個でよろしいですか？"
}
```

---

### Q8. 商品マスタの標準単位と一致している場合は？

**状況**: 卵を「3箱」と注文。商品マスタの標準単位は「箱」。過去も箱単位。

**正しい判断**: 
単位異常なし。顧客の通常パターンも「箱」なので確認不要。

**出力**:
```json
{
  "anomalies": [],
  "confirmation_needed": false
}
```

**関連**: Q6

### カテゴリC: 商品名の曖昧さ

### Q9. 「しめじ追加で」と言われたが、商品マスタに「しめじ」と「ぶなしめじ」がある場合は？

**状況**: Intake Agent が normalize_product した結果、候補が2つ以上存在。正規化で一意マッチしなかった。

**注文内容の例**:
> Intake Agent からの出力：
> ```json
> {
>   "products": [
>     {"name": "しめじ", "candidates": ["しめじ", "ぶなしめじ"], "quantity": null, "unit": null}
>   ]
> }
> ```

**正しい判断**: 
商品が一意に特定できない。severity = medium で候補を提示して確認。数量も指定されていない可能性もあるため、複数の異常がある場合は priority（商品特定）を優先。

**出力**:
```json
{
  "anomalies": [
    {
      "type": "ambiguous_product",
      "product": "しめじ",
      "severity": "medium",
      "candidates": ["しめじ", "ぶなしめじ"],
      "message": "商品が一意に特定できません"
    }
  ],
  "confirmation_needed": true,
  "confirmation_message": "しめじは「しめじ」と「ぶなしめじ」がございます。どちらでしょうか？"
}
```

**よくある間違い**:
- どちらかに勝手に決めてしまう

---

### Q10. 「トマト」と言われて、商品マスタに「トマト」と「フルーツトマト」がある場合は？

**状況**: normalize_product で「トマト」に一意マッチした場合 vs マッチしない場合。

**正しい判断**: 
- 一意マッチした場合：異常なし。確認不要。
- マッチしない場合（複数候補）：Q9と同じ。候補を提示して確認。

**出力（マッチした場合）**:
```json
{
  "anomalies": [],
  "confirmation_needed": false
}
```

**出力（複数候補の場合）**:
```json
{
  "anomalies": [
    {
      "type": "ambiguous_product",
      "product": "トマト",
      "severity": "medium",
      "candidates": ["トマト", "フルーツトマト"],
      "message": "商品が一意に特定できません"
    }
  ],
  "confirmation_needed": true,
  "confirmation_message": "トマトは「トマト」と「フルーツトマト」がございます。どちらでしょうか？"
}
```

### カテゴリD: 変更対象の不明確さ

### Q11. 「さっきのやつ減らして」のように、何を変更したいのかわからない場合は？

**状況**: 現在注文に3商品（りんご10箱、トマト10kg、卵3箱）。「さっきのやつ減らして」と来た。「やつ」が3商品のどれを指すか、新しい数量も不明。

**注文内容の例**:
> Intake Agent からの出力：
> ```json
> {
>   "instruction": "さっきのやつ減らして",
>   "current_order": [
>     {"name": "りんご", "quantity": 10, "unit": "箱"},
>     {"name": "トマト", "quantity": 10, "unit": "kg"},
>     {"name": "卵", "quantity": 3, "unit": "箱"}
>   ],
>   "target_product": null,
>   "new_quantity": null
> }
> ```

**正しい判断**: 
「やつ」が3商品のどれを指すか不明。変更したい数量も不明。2つの情報が必要なため、severity = medium で両方聞く（ただし1回の返信で1つずつ）。

**出力**:
```json
{
  "anomalies": [
    {
      "type": "ambiguous_target",
      "product": null,
      "severity": "medium",
      "message": "変更対象の商品と新しい数量が特定できません"
    }
  ],
  "confirmation_needed": true,
  "confirmation_message": "現在のご注文のうち、どの商品の数量を変更いたしましょうか？商品名と新しい数量をお知らせください。"
}
```

**よくある間違い**:
- 最後に追加した商品を「さっきの」と解釈して勝手に処理してしまう
- 「数量を減らして」の意味を確認せず、規定量（30%減等）で勝手に減らす

**状況**: 現在注文にトマト10kg。「トマト減らして」と来た。

**正しい判断**: 
商品は特定できたが新しい数量が不明。「どの位減らす？」を聞く必要がある。severity = medium。

**出力**:
```json
{
  "anomalies": [
    {
      "type": "ambiguous_quantity",
      "product": "トマト",
      "severity": "medium",
      "message": "変更後の数量が特定できません"
    }
  ],
  "confirmation_needed": true,
  "confirmation_message": "トマトは現在10kgのご注文ですが、何kgにご変更いたしましょうか？"
}
```

      "message": "数量が指定されていません"

}

### カテゴリE: 複数の異常が重なった場合

### Q14. 数量異常と単位異常が同時に起きた場合はどうする？

**状況**: トマト100箱（通常5kg単位で注文、しかも量が桁違い）。2つの異常がある。

**正しい判断**: 
severity が高い方（数量異常 = high）を先に1つだけ確認。1回の返信で複数の確認質問を投げない。顧客が答えやすくするため。もし数量が「100」ではなく「10」だった場合、単位異常（kg→箱）だけ確認すればよいため、順序が大事。

**出力（最初の確認）**:
```json
{
  "anomalies": [
    {
      "type": "quantity",
      "product": "トマト",
      "severity": "high",
      "message": "通常5kg前後のところ100箱のご注文です（単位も通常と異なります）"
    },
    {
      "type": "unit",
      "product": "トマト",
      "severity": "medium",
      "message": "通常はkg単位でのご注文ですが、箱単位です"
    }
  ],
  "confirmation_needed": true,
  "confirmation_message": "トマト100箱のご注文ですが、いつもは5kg前後です。数量をご確認いただけますか？",
  "next_question_if_confirmed": "その場合、単位はkg計算でよろしいですか？"
}
```

**よくある間違い**:
- 「数量も単位もおかしいです。100箱でいいですか？kg単位ではないですか？」と一度に聞いてしまう

---

### Q15. 異常を検知したが、顧客が意図的に大量注文している場合もあるのでは？

**状況**: イベント仕込みで通常の10倍量を注文。Zスコア > 3.0で high 判定。

**正しい判断**: 
確認は必須。ただし確認質問は「間違いだ」と決めつけず、「数量をご確認いただけますか？」と中立的に聞く。顧客が「これで OK」と言えば確定。中立的な言い方により、顧客の意図を尊重しつつリスクを低減できる。

**出力**:
```json
{
  "anomalies": [
    {
      "type": "quantity",
      "product": "りんご",
      "severity": "high",
      "message": "通常10箱前後のところ100箱のご注文です"
    }
  ],
  "confirmation_needed": true,
  "confirmation_message": "りんご100箱のご注文ですが、いつもは10箱前後です。数量をご確認いただけますか？"
}
```

**よくある間違い**:
- 「間違いですよね？」と決めつけた聞き方をしてしまう
- 確認せずに通してしまう

### カテゴリF: 確認不要のケース

### Q16. 全てが正常な注文の場合は？

**状況**: りんご10箱 + バナナ5kg。過去パターンと一致。単位も正常。数量も明確。

**正しい判断**: 
異常なし。確認不要でそのまま通す。確認しすぎると顧客体験が悪化するため、正常なケースは迅速に処理する。

**出力**:
```json
{
  "anomalies": [],
  "confirmation_needed": false
}
```

---

### Q17. 顧客が「OK」「はい」と肯定返信した場合は？

**状況**: pending_draft の確定。Exception Agent が「〇〇でいいですか？」と聞いて、顧客が「OK」と答えた。

**正しい判断**: 
肯定返信は異常検知の対象外。確認が既に終わったため、そのまま通す。この返信は Intake Agent に渡され、注文が「確定」状態に遷移する。

**出力**:
```json
{
  "anomalies": [],
  "confirmation_needed": false,
  "status": "confirmed"
}
```

## 5. 確認質問の書き方ルール

1. **丁寧語**（ですます調）
2. **1文で簡潔に**（長い説明は不要）
3. **具体的な選択肢を含める**（「どちらですか？」だけでなく候補を提示）
4. **顧客が1行で答えられる形にする**
5. **複数の異常がある場合は、severity が高い順に1つずつ確認する**（一度に全部聞かない）

## 6. やってはいけないこと

1. 異常を検知したのに確認せず通す
2. 確認質問なしで数量を勝手に修正する
3. 過去データが少ないのに異常判定する
4. 1回の返信で複数の確認質問を同時に投げる
5. 商品名の正規化をやり直す（Intake Agent が済ませている）
