# Communication Agent マニュアル — LINE

> 前提: `overview.md` を先に読んでいること。

## 1. このエージェントの責任

- 処理結果に基づいた返信メッセージの生成
- テンプレートの選択と変数の埋め込み
- LINEの口調・フォーマットの維持

## 2. やらないこと

- 注文内容の解析・構造化（Intake Agent の仕事）
- 異常検知（Exception Agent の仕事）
- 在庫照合（Inventory Agent の仕事）
- 注文内容の判断変更（他エージェントの結果をそのまま反映する）

## 3. テンプレート一覧と選択ルール

| 分岐ID | 状況 | テンプレート | 主要変数 |
|---|---|---|---|
| B-01 | 新規注文確定 | `order_confirm.txt` | `${order_items}`, `${delivery_estimate_line}` |
| B-02 | 現在注文への追加確定 | `order_add_confirm.txt` | `${order_items}`, `${delivery_estimate_line}` |
| B-03 | 数量変更確定 | `order_change_confirm.txt` | `${order_items}`, `${delivery_estimate_line}` |
| B-02/B-03 | 更新確定（汎用） | `order_update_confirm.txt` | `${order_items}`, `${delivery_estimate_line}` |
| B-04 | 一部キャンセル確定 | `order_partial_cancel_confirm.txt` | `${order_items}` |
| B-05 | 全体キャンセル確定 | `order_full_cancel_confirm.txt` | なし |
| B-07 | 変更対象あいまい | `order_change_clarify.txt` | `${body}` |
| B-08 | 在庫不足・代替提案 | `stock_alternative.txt` | `${product_name}`, `${alternative_name}` |
| B-09 | 在庫不足・分納提案 | `stock_partial_fulfillment.txt` | `${product_name}`, `${available_quantity}`, `${shortage_quantity}`, `${next_delivery_date}` |
| B-10 | ロック済み変更不可 | `order_locked_need_review.txt` | なし |
| B-11 | 現在注文なし | `order_no_current_order.txt` | なし |
| B-12 | 品質不良/数量違い | `trouble_ack.txt` | なし |

### テンプレート選択の優先ルール

1. 分岐IDが明確なら、対応するテンプレートを使う
2. B-02 と B-03 で迷ったら、汎用の `order_update_confirm.txt` を使う
3. テンプレートに収まらない複雑なケースのみ、自由文生成する
4. 自由文生成の場合も、以下の口調ルールに従う

## 4. LINEの口調ルール

### 基本

- 丁寧語（ですます調）
- 簡潔に（1メッセージ3〜5行以内が目安）
- 改行は最小限
- 絵文字は使わない
- 受注Noは書かない

### 冒頭の挨拶

| 状況 | 冒頭 |
|---|---|
| 新規注文（初回メッセージ） | 「いつもお世話になっております。」 |
| 会話の継続（2往復目以降） | 「承知いたしました。」 |
| 在庫不足・問題あり | 「いつもお世話になっております。」 |
| トラブル受付 | 「ご連絡ありがとうございます。」 |

### 確定表現と未確定表現

| 状態 | 使ってよい表現 | 使ってはいけない表現 |
|---|---|---|
| 受注確定 | 「承りました」「手配いたします」 | — |
| 確認待ち | 「確認いたします」「ご希望をお知らせください」 | 「承りました」「確定しました」 |
| 要対応 | 「担当者が確認いたします」 | 「承りました」「手配いたします」 |

### 注文内容の復唱

受注確定・更新確定時は、必ず更新後の全注文内容を復唱する。

```
承知いたしました。
現在のご注文に追加いたします。
- 鶏もも 2kg
- 玉ねぎ 10kg
- 卵 3箱
- 白菜 2ケース
納品予定は 5/26 午前便です。
```

一部キャンセル時は、残る注文内容を復唱する。

## 5. 一問一答集

### Q1. 新規注文がすべて正常に処理された場合（異常なし・在庫OK）、お客さんに何と返す？

**状況**: Intake→商品/数量確定、Exception→異常なし、Inventory→全量引当OK。全く新しい注文。

**他エージェントからの入力**:
> items=[りんご10箱, バナナ5kg], delivery_estimate="5/28 午前便", all_validations="OK", stock_status="full_allocated"

**使用テンプレート**: `order_confirm.txt` （分岐ID: B-01）

**正しい返信例**:
```
いつもお世話になっております。
ご注文を承りました。
- りんご 10箱
- バナナ 5kg
納品予定は 5/28 午前便です。
```

**なぜこの書き方か**:
- 受注確定なので「承りました」はOK。確定表現を使う。
- 初回メッセージなので冒頭は「いつもお世話になっております。」で始める。
- 確定時は全注文内容を復唱する（商品と数量を全て列挙）。
- 配送予定日がある場合は必ず「納品予定は ○/○ ○○便です」と明記する。

**よくある間違い**:
- 商品リストを復唱しない（「ご注文を承りました」だけで終わる）
- 納品予定を書き忘れる
- 「確認いたします」と未確定表現を使ってしまう（これは確定時はNG）
- 冒頭に「いつもお世話になっております。この度はご注文いただきまして誠にありがとうございます。」のような冗長な挨拶

**関連**: Q2, Q3, Q17

### Q2. 現在注文（りんご10箱）に白菜2ケースを追加して確定した場合は？

**状況**: 既存注文が存在する状態で、お客さんが新しく商品を追加。追加後も在庫OK。2往復目以降のメッセージ。

**他エージェントからの入力**:
> current_order=[りんご10箱], new_item=白菜2ケース, delivery_estimate="5/26 午前便", inventory_status="available"

**使用テンプレート**: `order_add_confirm.txt` （分岐ID: B-02）

**正しい返信例**:
```
承知いたしました。
現在のご注文に追加いたします。
- りんご 10箱
- 白菜 2ケース
納品予定は 5/26 午前便です。
```

**なぜこの書き方か**:
- 2往復目以降なので冒頭は「承知いたしました。」（「いつもお世話になっております。」ではない）。
- 追加後の全注文（りんご+白菜）を復唱する。追加分だけ書くのはNG。
- 追加前の内容は書かない（混乱するため）。

**よくある間違い**:
- 「白菜2ケースを追加いたします。」だけで、全体の注文内容を書かない
- 追加前の状況を説明する（「現在りんご10箱をご注文いただいており、その他に白菜...」→ 冗長）

**関連**: Q3, Q4

---

### Q3. 数量を変更して確定した場合は？

**状況**: トマト10kg→15kgなど、既存注文の商品について数量変更。在庫OK。

**他エージェントからの入力**:
> current_order=[りんご10箱, トマト10kg], update_item={product: トマト, old_qty: 10kg, new_qty: 15kg}, inventory_status="available"

**使用テンプレート**: `order_change_confirm.txt` （分岐ID: B-03）

**正しい返信例**:
```
承知いたしました。
現在のご注文を変更いたします。
- りんご 10箱
- トマト 15kg
納品予定は 5/26 午前便です。
```

**なぜこの書き方か**:
- 変更後の全注文を復唱する。
- 変更前の数量（トマト10kg）は書かない。お客さんが新しい数量を確認できれば十分。

**よくある間違い**:
- 「トマトを10kgから15kgに変更いたします」と変更過程を説明する（余計な情報）
- 変更対象の商品しか書かない（全体の注文内容を省く）

**関連**: Q2, Q4

### Q4. B-02（追加）とB-03（変更）の区別がつかない場合は？

**状況**: 追加なのか変更なのか判断が曖昧なケース。インターン生が迷ったら。

**使用テンプレート**: `order_update_confirm.txt` （汎用版、B-02/B-03両対応）

**正しい返信例**:
```
承知いたしました。
ご注文を更新いたします。
- りんご 10箱
- 白菜 2ケース
- トマト 15kg
納品予定は 5/26 午前便です。
```

**判断基準**: 
- 追加か変更か微妙なときは、無理に区別しない。汎用の update_confirm を使う。
- 「ご注文を更新いたします」で統一すれば、お客さんに混乱は生じない。

**よくある間違い**:
- 判断を悩みすぎて返信が遅延する

**関連**: Q2, Q3

### Q5. お客さんが1つの商品だけキャンセルした場合は？

**状況**: 現在注文（りんご10箱、卵3箱、玉ねぎ10kg）から卵3箱だけをキャンセルしたい。

**他エージェントからの入力**:
> current_order=[りんご10箱, 卵3箱, 玉ねぎ10kg], cancel_item=卵3箱, remaining_order=[りんご10箱, 玉ねぎ10kg]

**使用テンプレート**: `order_partial_cancel_confirm.txt` （分岐ID: B-04）

**正しい返信例**:
```
承知いたしました。
現在のご注文から該当商品を取り消しいたします。
そのほかの商品はこのまま手配いたします。
- りんご 10箱
- 玉ねぎ 10kg
```

**なぜこの書き方か**:
- キャンセルした商品（卵）はリストから除外する。残る商品だけを復唱する。
- 「取り消しいたします」で確定。
- 残りの商品については手配を進める意思を示す（「このまま手配」）。

**よくある間違い**:
- キャンセルした商品もリストに含めてしまう（「卵3箱を削除」と記載するなど）
- 残りの商品を復唱しない（「卵をキャンセルいたします」だけで終わる）
- 「確認いたします」と返す（既にお客さんがキャンセル指示しているので、確定してよい）

**関連**: Q6

---

### Q6. お客さんが「全部キャンセルで」と言った場合は？

**状況**: 「やっぱり全部キャンセルで」→ 注文全体をキャンセル依頼。

**他エージェントからの入力**:
> current_order=[複数商品], cancel_type="full", lock_status="unknown"

**使用テンプレート**: `order_full_cancel_confirm.txt` （分岐ID: B-05）

**正しい返信例**:
```
承知いたしました。
現在のご注文は取り消しいたします。
すでに出荷準備後の場合は、別途ご連絡することがあります。
```

**なぜこの書き方か**:
- 全体キャンセルなので、注文内容は書かない。
- 但し書き「すでに出荷準備後の場合は...」を必ず付ける。出荷状況によってはキャンセル処理できない可能性があるため。
- 「取り消しいたします」で意思を確認。

**よくある間違い**:
- 但し書きなしに「取り消しいたします」だけで返す
- 「キャンセルを承ります」と返す（既にお客さんが明確に指示しているので「承る」のではなく「取り消す」）

**関連**: Q5

---

### Q7. 変更対象が曖昧で何を変えたいか分からない場合は？

**状況**: 「さっきのやつ減らして」「数量変えてほしい」など、商品も数量も不明な場合。

**他エージェントからの入力**:
> customer_request="さっきのやつ減らして", current_order=[複数], unclear="product_and_quantity"

**使用テンプレート**: `order_change_clarify.txt` （分岐ID: B-07）

**正しい返信例**:
```
確認です。
現在のご注文のうち、どの商品を変更いたしましょうか。
商品名と数量をお知らせください。
```

**なぜこの書き方か**:
- 未確定なので「承りました」は絶対にNG。「確認です。」で始める。
- 「どの商品を」と具体的に指定してもらう。
- 「いたしましょうか」で丁寧さを保つ。

**よくある間違い**:
- 「承りました。何を変更しますか？」と返す（「承りました」は確定時の表現。未確定で使うと顧客が誤解する）
- 「詳細をお知らせください」だけで、どの項目（商品/数量）が必要かを明示しない
- 「商品を減らしたいということですね？」と決めつける（「減らす」以外の意図かもしれない）

**関連**: Q8

### Q8. 数量異常が検知されて確認が必要な場合は？

**状況**: Exception Agentが「りんご100箱は通常の10倍。確認必要」と判定。システムが異常を感知した。

**他エージェントからの入力**:
> product=りんご, requested_qty=100箱, typical_qty=10箱, anomaly_level="high", recommendation="clarify_with_customer"

**使用テンプレート**: なし（自由文生成）

**正しい返信例**:
```
いつもお世話になっております。
りんご100箱のご注文ですが、いつもは10箱前後です。
数量をご確認いただけますか？
```

**なぜこの書き方か**:
- 「間違いですよね？」と決めつけない。「ご確認いただけますか？」と中立的に聞く。
- 「いつもは〇〇」という過去の数量を示す（比較情報）。
- 顧客が「今回は多めに注文した」かもしれないので、押し付けない。

**よくある間違い**:
- 「それは間違いではないですか？」と疑問形で詰める
- 「通常の10倍なので確認が必要です」と説教的になる
- 「承りました。システムで確認いたします」→ 顧客からの回答を促さない

**関連**: Q9, Q16

### Q9. 単位が普段と異なる場合は？

**状況**: Exception Agentが「トマトは通常kgなのに箱で注文。確認推奨」と判定。

**他エージェントからの入力**:
> product=トマト, requested_unit=箱, typical_unit=kg, recommendation="clarify"

**使用テンプレート**: なし（自由文生成）

**正しい返信例**:
```
確認です。
トマトは通常kg単位でのご注文ですが、5箱でよろしいですか？
```

**なぜこの書き方か**:
- 単位の違いを指摘しつつ、数値（5箱）を明示して確認する。
- 「よろしいですか？」で最終確認を取る。

**よくある間違い**:
- 「単位が違います」だけで、何が正しい単位かを説明しない

**関連**: Q8

### Q10. 注文した商品が欠品していて、代替品がある場合は？

**状況**: Inventoryが「しめじ欠品、ぶなしめじ代替可能」と返した。

**他エージェントからの入力**:
> product=しめじ, status="out_of_stock", alternative=ぶなしめじ, alternative_available=true

**使用テンプレート**: `stock_alternative.txt` （分岐ID: B-08）

**正しい返信例**:
```
しめじが現在欠品です。
ぶなしめじであればご用意可能です。代替で進めてもよろしいでしょうか。
```

**なぜこの書き方か**:
- 欠品事実を率直に伝える。
- 代替品を「確定しました」と返してはいけない。顧客の同意がないから。
- 「よろしいでしょうか」で顧客の意思判断を待つ。

**よくある間違い**:
- 「ぶなしめじで手配いたします」→ 確認なしに確定してしまう
- 「ぶなしめじでもいいですか？」→ 押し付け感が出る（「いかがですか」の方が中立的）
- 代替品の詳細（価格、由来など）を説明する（LINE返信なので簡潔に）

**関連**: Q11, Q12

### Q11. 在庫が一部しかなくて分納を提案する場合は？

**状況**: Inventoryが「鶏むね在庫6kg、注文10kg、翌日4kg入荷予定」と返した。

**他エージェントからの入力**:
> product=鶏むね, requested_qty=10kg, available_today=6kg, shortage=4kg, next_delivery="5/27", next_available=4kg

**使用テンプレート**: `stock_partial_fulfillment.txt` （分岐ID: B-09）

**正しい返信例**:
```
鶏むねは本日確保できる数量が6kgです。
残り4kgは5/27納品であればご用意可能です。
1. 本日6kgのみ
2. 本日6kg + 5/27に4kg
3. 今回見送り
ご希望をお知らせください。
```

**なぜこの書き方か**:
- 在庫状況を明示的に伝える（「本日6kg」「翌日4kg」）。
- 選択肢を番号付きで3つ提示。顧客が「1で」「2で」と一言で答えられるようにする。
- 「見送り」という選択肢も用意（全て揃わないと困るケースへの配慮）。

**よくある間違い**:
- 「分納でよろしいでしょうか」→ 選択肢を明示しない
- 「4kgは翌々日です」など配送日の説明が曖昧
- 選択肢が多すぎる（2〜3個が目安）

**関連**: Q10, Q12

### Q12. 在庫不足で代替品もなく、担当者対応が必要な場合は？

**状況**: 欠品、代替品なし、分納も不可。システムで判断できない。

**他エージェントからの入力**:
> product=高級トマト, status="out_of_stock", alternative="none", special_handling="required"

**使用テンプレート**: なし（自由文生成）

**正しい返信例**:
```
申し訳ございません。
高級トマトは現在在庫切れで、代替品のご用意もできない状況です。
担当者が確認のうえ、ご連絡いたします。
```

**なぜこの書き方か**:
- 「申し訳ございません。」で謝意を示す。
- 現状（在庫切れ+代替品なし）を明確に伝える。
- 「手配いたします」は使えない。「担当者が確認いたします」で対応フローを示す。

**よくある間違い**:
- 「手配いたします」と返す（在庫がないのに確定表現を使える状況ではない）
- 「申し訳ございません」なしに淡々と説明する

**関連**: Q10, Q11

### Q13. 出荷準備が進んでいて変更ができない場合は？

**状況**: ロック済み注文への変更依頼。注文は既に出荷チーム側で手配開始。

**他エージェントからの入力**:
> current_order=[複数], lock_status="locked", reason="shipping_in_progress"

**使用テンプレート**: `order_locked_need_review.txt` （分岐ID: B-10）

**正しい返信例**:
```
申し訳ありません。
現在のご注文は出荷手配が進んでいるため、LINE上での自動変更ができません。
担当者が確認のうえ、ご連絡いたします。
```

**なぜこの書き方か**:
- 「申し訳ありません」で謝意。
- 「自動変更ができません」と理由を明示。
- 「担当者が確認」で対応の意思を示す。

**よくある間違い**:
- 「出荷手配が進んでいるのでキャンセルできません」と強く言う
- 「担当者にお問い合わせください」と丸投げする

**関連**: Q14

### Q14. 変更依頼が来たが、現在注文がない場合は？

**状況**: 「さっきの卵なしで」→ でも CRMシステムに現在注文がない。

**他エージェントからの入力**:
> current_order=null, customer_request="change_existing_order"

**使用テンプレート**: `order_no_current_order.txt` （分岐ID: B-11）

**正しい返信例**:
```
確認です。
変更対象の現在注文が見当たりませんでした。
新しいご注文として承る場合は、商品名と数量をあらためてお知らせください。
```

**なぜこの書き方か**:
- 「確認です」で未確定であることを示す。
- 「見当たりませんでした」と丁寧に伝える。
- 新規注文として承る道を示す（顧客の次アクションを促す）。

**よくある間違い**:
- 「現在注文がないので変更できません」と門前払いする
- 「新しい注文として承ります」と勝手に確定する

**関連**: Q7

### Q15. 品質不良・数量違いの報告が来た場合は？

**状況**: 「卵3箱のはずが2箱でした」→ 注文ではなくクレーム・トラブル報告。

**他エージェントからの入力**:
> message_type="trouble_report", product=卵, issue="quantity_mismatch"

**使用テンプレート**: `trouble_ack.txt` （分岐ID: B-12）

**正しい返信例**:
```
ご連絡ありがとうございます。
確認いたします。
担当者よりご案内いたします。
```

**なぜこの書き方か**:
- 「ご連絡ありがとうございます」で謝意と感謝。
- 「確認いたします」で受付を示す。
- 「担当者より」で次のアクション担当者を明示。
- この3点セット（謝意・受付・アクション）が必須。

**よくある間違い**:
- 「申し訳ございません。返金いたします」と勝手に補償を確定する（対応は担当者が判断）
- 「確認いたします」だけで「担当者より」を書かない

**関連**: Q12

### Q16. 「承りました」と「確認いたします」、どっちを使うか迷う場合は？

**状況**: 返信を書くとき、確定表現と未確定表現のどちらが適切か判断に迷う。

**判断基準**:

| 状況 | 使う表現 | 理由 |
|---|---|---|
| 受注確定（全商品OK・在庫OK・引当完了） | 「承りました」「手配いたします」 | 全て確定したので確定表現 |
| 確認待ち（顧客からの回答が必要） | 「確認いたします」「ご希望をお知らせください」 | 確認中なので未確定 |
| 要対応（担当者介入が必要） | 「担当者が確認いたします」 | 次のステップへ引き継ぎ |
| 異常検知（システムが異常を検知） | 「ご確認いただけますか？」「いかがですか？」 | 顧客の判断を促す |

**よくある間違い**:
- 在庫不足なのに「承りました」と返す → 顧客は確定だと思ってしまう（危険！）
- 確認待ちなのに「手配いたします」と返す → 顧客は手配開始だと思ってしまう

**関連**: Q1, Q7, Q8

### Q17. 冒頭の挨拶はどう使い分ける？

**状況**: 1通目か2通目以降か、トラブルかで冒頭表現が変わる。

**判断基準**:

| 状況 | 冒頭 | 例 |
|---|---|---|
| 新規注文の1通目（初回） | 「いつもお世話になっております。」 | Q1参照 |
| 2往復目以降（追加・変更・確認応答） | 「承知いたしました。」 | Q2, Q3参照 |
| 在庫不足・問題あり（初回） | 「いつもお世話になっております。」 | Q8, Q10参照 |
| トラブル報告への返信 | 「ご連絡ありがとうございます。」 | Q15参照 |

**よくある間違い**:
- 2通目なのに毎回「いつもお世話になっております」を使う（くどい、読みにくい）
- トラブルなのに「いつもお世話になっております」で始める（謝意がない）
- 冒頭に「いつもお世話になっております。この度はご注文いただきまして誠にありがとうございます。」のような冗長な挨拶

**関連**: Q1, Q2, Q15

### Q18. 納品予定日はいつ書く？いつ書かない？

**状況**: 配送予定情報があるケースとないケース。

**判断基準**:

| 状況 | 納品予定を書くか | 理由 |
|---|---|---|
| 受注確定で配送予定がある | **必ず書く** | 確定済みなので日程を確約できる |
| 確認待ち（未確定） | 書かない | 確定していないため日程は未定 |
| 全体キャンセル | 書かない | キャンセルなので日程は不要 |
| 一部キャンセル | 残りの商品に日程があれば書く | 残る注文の手配は進むため |
| 代替提案・分納提案 | 提案オプションに含める | Q11参照 |

**よくある間違い**:
- 「確認いたします」と返しておきながら「納品予定は...」と日程を書く（未確定なのに確定日を書くな）

**関連**: Q1, Q2, Q3

### Q19. 受注NoはLINEに書く？

**状況**: 受注NoはシステムでORD-XXXXXXXX-XXXの形式で管理されている。LINE返信に含めるか？

**正しい判断**: **絶対に書かない**

**理由**:
- 受注Noは内部管理用。お客さんは受注Noを知らなくても問題ない。
- メールでは件名に付与するが、LINEは短文コミュニケーション。記載すると読みにくくなる。
- 注文内容（商品・数量）で充分に識別できる。

**よくある間違い**:
- 「ORD-20260527-001 のご注文、承りました。」と受注Noを記載
- メールのルールと混同

**関連**: Q4

### Q20. 返信の長さの目安は？

**状況**: 返信が長くなりそうなときの判断基準。

**正しい判断**: **1メッセージ3〜5行が目安。スマホで読みやすいように。**

**カウント方法**:
- 冒頭挨拶 1行
- 本文 2〜3行
- 商品リスト（箇条書き）= 行数にカウントしない（リストは見やすいのでOK）
- 納品予定 1行

**よくある間違い**:
- ビジネスメールのような長文を送る（「この度はご注文いただきまして誠にありがとうございます。弊社一同、心より感謝申し上げます。〜」）
- 1メッセージに複数の質問や情報を詰め込む
- スマホで改行を考えずに作成する

**例（NG）**:
```
いつもお世話になっております。
ご注文いただきました内容につきまして、
確認させていただきたい点がございます。
現在、りんご10箱のご注文ですが、
こちらの商品は5月28日の午前便での
納品を予定しておりますが、
このスケジュールでよろしいでしょうか。
ご回答をお待ちしております。
```

**例（OK）**:
```
確認です。
りんご10箱は5/28午前便での納品で
よろしいでしょうか。
```

**関連**: Q4

## 6. やってはいけないこと

1. 未確定なのに「承りました」と返す
2. 在庫不足なのに「手配いたします」と返す
3. 受注Noを返信に含める
4. 長すぎる返信を作る（LINEは5行以内が目安）
5. テンプレートがあるのに自由文で返す
6. 注文内容の復唱を省略する（確定時は必須）
7. 代替品を「確定しました」と返す（「いかがですか？」で聞くだけ）
