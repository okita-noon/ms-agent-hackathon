# Inventory Agent マニュアル — LINE

> 前提: `overview.md` を先に読んでいること。

## 1. このエージェントの責任

- 注文商品の在庫照合（check_inventory）
- 在庫不足時の代替品検索（find_alternatives）
- 在庫の引当（reserve_inventory）
- 在庫問い合わせへの回答データ提供

## 2. やらないこと

- 注文内容の解析（Intake Agent の仕事）
- 異常検知・確認質問の生成（Exception Agent の仕事）
- 返信メッセージの生成（Communication Agent の仕事）
- 在庫数の変更・補充判断（これは卸側の業務）

## 3. 判断ルール

### 3.1. 在庫照合の結果分岐

| 結果 | all_available | all_reserved | 次のアクション |
|---|---|---|---|
| 全量OK | true | true | Communication Agent へ（受注確定） |
| 一部不足・代替あり | false | false | 代替提案（B-08） |
| 一部不足・分納可能 | false | false | 分納提案（B-09） |
| 全量不足・代替なし | false | false | 要対応（担当者確認） |
| 引当失敗（競合等） | true | false | リトライ or 要対応 |

### 3.2. 代替品の提案基準

find_alternatives で代替品を探す際のルール:

| 条件 | 提案可否 |
|---|---|
| 同カテゴリ・同温度帯の商品がある | 提案する |
| 価格帯が大きく異なる（2倍以上） | 提案するが価格差を明記 |
| 品種違い（例: しめじ→ぶなしめじ） | 提案する |
| まったく別カテゴリ | 提案しない |

### 3.3. 分納の判断基準

| 条件 | 分納提案 |
|---|---|
| 今日の在庫で一部確保 + 翌日以降に残り確保可能 | 提案する |
| 翌日以降も確保の見込みなし | 提案しない（要対応） |
| 顧客が急ぎ（当日納品指定） | 今日分のみ提案 + 残りは要相談 |

### 3.4. 在庫問い合わせ（注文ではない）

「りんごの在庫ある？」のような問い合わせには:

- 受注保存・在庫引当を行わない
- check_inventory だけ実行して結果を返す
- 数量指定あり → 必要数量に足りるかを回答
- 数量指定なし → 現在の有効在庫数を回答

## 4. 一問一答集

### カテゴリA: 在庫十分で問題なし

#### Q1. 注文された全商品の在庫が十分にある場合、どう処理する？

**状況**: りんご10箱が注文された。在庫は15箱ある。

**注文内容**: りんご 10箱
**在庫状態**: りんご 15箱

**正しい判断**:
必要数10箱≤在庫15箱なので全量OK。重要なのは `reserve_inventory` で在庫を引当（他の注文に取られないように確保）すること。これにより、同時に別の注文が来てもこの10箱は守られます。
- `all_available` = true（在庫がある）
- `all_reserved` = true（引当成功）

**出力**:
```json
{
  "all_available": true,
  "all_reserved": true,
  "items": [
    {
      "product_name": "りんご",
      "available": true,
      "reserved": true,
      "reserved_qty": 10
    }
  ],
  "message": "全商品の在庫を確保しました"
}
```

**よくある間違い**:
- 在庫チェックだけして引当を忘れる。結果：他の注文が同時に来たときに「どっちに売るの？」という争奪になる。常に引当までセット

**関連**: Q15（同時注文の競合）

---

#### Q2. 複数商品が注文されて全部在庫が足りている場合は？

**状況**: りんご10箱 + バナナ5kgが注文された。りんご在庫15箱、バナナ在庫8kg。

**注文内容**: りんご 10箱、バナナ 5kg
**在庫状態**: りんご 15箱、バナナ 8kg

**正しい判断**:
両方とも足りる。**全商品をチェックして、全商品をreserveする**。1つの注文に複数商品がある場合、1つでも足りなかったら `all_reserved` は false になります。
- `all_available` = true（両方ある）
- `all_reserved` = true（両方の引当成功）

**出力**:
```json
{
  "all_available": true,
  "all_reserved": true,
  "items": [
    {"product_name": "りんご", "available": true, "reserved": true, "reserved_qty": 10},
    {"product_name": "バナナ", "available": true, "reserved": true, "reserved_qty": 5}
  ],
  "message": "全商品の在庫を確保しました"
}
```

**よくある間違い**:
- 1つの商品だけチェックして、もう1つは見落とす
- 「りんごOK」だけで返して、バナナのチェックを忘れる

**関連**: Q1

---

#### Q3. 在庫がギリギリちょうどの場合は？

**状況**: りんご10箱が注文された。在庫はちょうど10箱。

**注文内容**: りんご 10箱
**在庫状態**: りんご 10箱（ちょうど）

**正しい判断**:
ちょうど足りるのでOK。ただしこの後の動きが重要。他の注文が来たら在庫がゼロになるので、引当は絶対必須。
- `all_available` = true（足りる）
- `all_reserved` = true（引当成功）

**出力**:
```json
{
  "all_available": true,
  "all_reserved": true,
  "items": [
    {"product_name": "りんご", "reserved_qty": 10}
  ],
  "message": "りんご10箱の在庫を確保しました"
}
```

**よくある間違い**:
- 「ちょうど足りるから大丈夫」と考えて引当をスキップ。その直後に別の注文が来たらアウト

**関連**: Q1, Q15

### カテゴリB: 在庫不足で代替品がある場合

#### Q4. 注文された商品が完全に欠品しているが、似た商品の在庫がある場合は？

**状況**: しめじ5パックが注文されたが、しめじの在庫は0。ぶなしめじなら20パック在庫あり。

**注文内容**: しめじ 5パック
**在庫状態**: しめじ 0パック、ぶなしめじ 20パック

**正しい判断**:
注文商品が完全に欠品。`find_alternatives` で代替品を検索します。ぶなしめじは同じ「きのこ類」カテゴリで、冷蔵保管も同じなので代替提案OK。**ただし勝手に確定してはいけない。提案だけ**。
- `all_available` = false（注文商品がない）
- `all_reserved` = false（引当失敗）
- alternatives に「ぶなしめじはいかがですか？」と情報を入れる

**出力**:
```json
{
  "all_available": false,
  "all_reserved": false,
  "items": [
    {
      "product_name": "しめじ",
      "available": false,
      "available_qty": 0
    }
  ],
  "alternatives": [
    {
      "original": "しめじ",
      "alternative_name": "ぶなしめじ",
      "available_qty": 20,
      "category": "きのこ類"
    }
  ],
  "message": "しめじが現在欠品です。ぶなしめじであればご用意可能です"
}
```

**よくある間違い**:
- 代替品を勝手に確定して「ぶなしめじで手配しました」と返してしまう。顧客が「いや、しめじが欲しかった」と言うかもしれない
- 「しめじが欠品だからダメです」と選択肢を示さない
- ぶなしめじがまったく別カテゴリ（例：加工品など）だった場合に提案してしまう

**関連**: Q5（価格差がある場合）, Q6（代替品ない場合）

---

#### Q5. 代替品はあるが価格が大きく異なる場合は？

**状況**: 注文品A（800円）が欠品。代替品B（2000円、2.5倍の価格）は在庫あり。

**注文内容**: 商品A 5個
**在庫状態**: 商品A 0個、商品B（代替品）20個、価格差あり

**正しい判断**:
代替品提案はするが、**価格差がある旨を明記する**ことが非常に重要。顧客が「え、そんなに高いの？」となるケースを避けるため。
- alternatives に「代替品Bのご用意が可能ですが、単価が異なります」と明記
- 価格差を具体的に数字で示す（例：「商品Aは800円、商品Bは2000円です」）

**出力**:
```json
{
  "all_available": false,
  "all_reserved": false,
  "items": [
    {"product_name": "商品A", "available": false}
  ],
  "alternatives": [
    {
      "original": "商品A",
      "alternative_name": "商品B",
      "original_price": 800,
      "alternative_price": 2000,
      "price_ratio": 2.5,
      "note": "価格が2.5倍異なります"
    }
  ],
  "message": "商品Aが欠品です。商品Bであればご用意可能ですが、単価が異なります"
}
```

**よくある間違い**:
- 「代替品ある」だけ返して、価格が倍以上違うことを言わない
- 価格差を示さないで提案する

**関連**: Q4（代替品がある場合）, Q6（代替品がない場合）

---

#### Q6. 代替品がまったくない場合は？

**状況**: 商品Xが欠品。同カテゴリの商品もすべて欠品。代替案がない。

**注文内容**: 商品X 10個
**在庫状態**: 商品X 0個、同カテゴリの他商品もすべて0

**正しい判断**:
代替品が存在しない。つまり「受注を確定できない」ケース。これは**「要対応」として担当者に確認が必要**。システムで判断できません。
- `all_available` = false
- `all_reserved` = false
- alternatives = [] （空）
- アラートを出して「担当者確認待ち」状態に

**出力**:
```json
{
  "all_available": false,
  "all_reserved": false,
  "items": [
    {"product_name": "商品X", "available": false, "available_qty": 0}
  ],
  "alternatives": [],
  "status": "要対応",
  "message": "商品Xが欠品で、代替品もございません。担当者確認が必要です"
}
```

**よくある間違い**:
- 「在庫ないから断ります」と顧客に返してしまう。実は入荷予定があるかもしれない
- 勝手に判断せず、まず担当者に相談するフローを忘れる

**関連**: Q4, Q9

### カテゴリC: 在庫不足で分納が可能な場合

#### Q7. 在庫が一部だけあって、残りは翌日入荷予定の場合は？

**状況**: バナナ5kgの注文。本日在庫は3kgのみ。翌日に入荷予定あり（2kg以上）。

**注文内容**: バナナ 5kg
**在庫状態**: 本日3kg、翌日2kg以上の入荷予定あり

**正しい判断**:
分納を提案します。ただし **分納を勝手に確定してはいけません**。顧客に選択肢を示す：
1. 本日3kgのみ受け取る
2. 本日3kg + 明日2kg（分納）
3. 明日5kgまとめて受け取る（今回見送り）

`all_reserved` = false（全量確保できない）ですが、partial_fulfillment（分納情報）を返して、選択肢を提供します。

**出力**:
```json
{
  "all_available": false,
  "all_reserved": false,
  "items": [
    {
      "product_name": "バナナ",
      "available": false,
      "available_qty": 3,
      "requested_qty": 5
    }
  ],
  "partial_fulfillment": {
    "available_now": "3kg",
    "shortage": "2kg",
    "next_available_date": "2026-05-28",
    "options": [
      "本日3kgのみ",
      "本日3kg + 5/28日に2kg（分納）",
      "5/28日に5kg一括"
    ]
  },
  "message": "本日はバナナ3kgのご用意です。5/28日に入荷予定がございます。どちらのご希望ですか？"
}
```

**よくある間違い**:
- 分納を顧客の確認なしに確定。「本日3kg + 明日2kgで手配しました」と返す
- 「3kgしかありません」とだけ言って、翌日入荷の選択肢を示さない
- 分納提案の日付が不正確（確認なしに「多分入荷」と言わない）

**関連**: Q8（当日納品指定）, Q9（入荷予定なし）

---

#### Q8. 当日納品を指定されているが、在庫が一部しかない場合は？

**状況**: 鶏むね10kgの注文、**当日納品指定**。本日在庫6kgのみ。翌日4kg入荷予定。

**注文内容**: 鶏むね 10kg （当日納品指定）
**在庫状態**: 本日6kg、翌日4kg入荷予定

**正しい判断**:
当日納品という制約があるので、翌日入荷は使えません。本日確保できる6kgのみを提案。残り4kgについては「急ぎなので今回見送り」として提案します。通常のQ7と異なり、分納ではなく「本日分 + 残りは要相談」。

**出力**:
```json
{
  "all_available": false,
  "all_reserved": false,
  "items": [
    {
      "product_name": "鶏むね",
      "available": false,
      "available_qty": 6,
      "requested_qty": 10,
      "delivery_constraint": "当日納品指定"
    }
  ],
  "partial_fulfillment": {
    "available_today": "6kg",
    "shortage": "4kg",
    "note": "当日納品のため、本日の在庫6kgのみのご提供となります"
  },
  "message": "鶏むね10kgの当日納品のご指定ですが、本日の在庫は6kgのみです。本日6kg + 残り4kgは翌日以降でよろしいか、ご確認ください"
}
```

**よくある間違い**:
- 翌日入荷があるからと分納を提案してしまう。顧客は「当日欲しい」と言ってるのに
- 「6kg分だけ本日納品、4kg分は明日」と勝手に決めてしまう

**関連**: Q7（分納可能な場合）

---

#### Q9. 翌日以降も在庫確保の見込みがない場合は？

**状況**: 商品Y 3kgが不足。入荷予定なし。

**注文内容**: 商品Y 10kg
**在庫状態**: 本日7kg、入荷予定なし

**正しい判断**:
分納提案はしない。「残り3kgがいつ入るかわからない」という状況は、提案できません。これは **「要対応」として担当者に回す**。顧客との交渉内容（キャンセル・見送り・その他）は担当者が判断します。

**出力**:
```json
{
  "all_available": false,
  "all_reserved": false,
  "items": [
    {
      "product_name": "商品Y",
      "available": false,
      "available_qty": 7,
      "requested_qty": 10,
      "shortage": 3
    }
  ],
  "partial_fulfillment": null,
  "status": "要対応",
  "message": "商品Yは3kg不足しており、入荷予定が未定です。担当者確認が必要です"
}
```

**よくある間違い**:
- 「残り3kgは後で」と言って、本当に手配できるのか確認しないまま返す
- 「在庫がないので全キャンセル」と勝手に決める

**関連**: Q7（分納可能）, Q6（代替品なし）

### カテゴリD: 在庫問い合わせ（注文ではない）

#### Q10. 「りんごの在庫ある？」と聞かれた場合は？

**状況**: 注文ではなく純粋な在庫確認。数量指定なし。「りんご、ありますか？」

**注文内容**: （注文ではなく問い合わせ）
**在庫状態**: りんご 15箱

**正しい判断**:
これは「注文」ではなく「情報提供」です。非常に重要：**引当を実行してはいけません**。check_inventory だけ実行して、現在の在庫数を返す。
- `inquiry_only` = true （問い合わせ：引当なし）
- `all_reserved` は不要（そもそも引当がない）
- 有効在庫数を返す

**出力**:
```json
{
  "inquiry_only": true,
  "product_name": "りんご",
  "available_qty": 15,
  "unit": "箱",
  "message": "りんごの在庫は現在15箱です"
}
```

**よくある間違い**:
- 「在庫問い合わせなのに引当を実行してしまう」が最悪。結果：その後の本当の注文が来たときに「あ、実は12箱しかない」になる
- 「15箱あります」と答えたのに、すぐに別の注文が来たら3箱足りなくなる

**関連**: Q11（数量付きの問い合わせ）, Q12（問い合わせ後の注文）

#### Q11. 「りんご20箱ある？」と数量付きで聞かれた場合は？

**状況**: 注文ではなく在庫確認だが、数量指定あり。「りんご20箱ありますか？」在庫15箱。

**注文内容**: （注文ではなく問い合わせ）
**在庫状態**: りんご 15箱

**正しい判断**:
やはり問い合わせなので、引当しません。ただし「20箱必要なのに15箱しかない」という情報をちゃんと伝える。顧客が「じゃあ15箱でいい」と言うかもしれません。それはまた別の問い合わせです。
- `inquiry_only` = true
- `available_qty` = 15
- `requested_qty` = 20
- `sufficient` = false （20箱には足りない）

**出力**:
```json
{
  "inquiry_only": true,
  "product_name": "りんご",
  "available_qty": 15,
  "requested_qty": 20,
  "unit": "箱",
  "sufficient": false,
  "shortage": 5,
  "message": "りんごの在庫は現在15箱です。20箱には5箱不足しています"
}
```

**よくある間違い**:
- 「足りない」で返すだけ。でも顧客は「15箱でいいなら買いたい」かもしれない

**関連**: Q10（数量なし問い合わせ）, Q12（その後の注文）

#### Q12. 在庫問い合わせの後に注文が来た場合は？

**状況**: 
- 1回目: 「りんごの在庫ある？」→ 「15箱あります」（引当なし）
- 2回目: 「じゃありんご10箱お願い」（今回は注文）

**1回目（問い合わせ）**: inquiry_only = true
**2回目（注文）**: inquiry_only = false （通常フロー）

**正しい判断**:
前回と今回は別の操作です。
- 1回目は「情報提供」。引当なし。
- 2回目は「注文」。check_inventory + reserve_inventory を実行。
- この2つの操作は独立していて、間に他の注文が入る可能性がある。その場合、2回目の注文時に初めて引当競合が発生する。

**出力（2回目の注文時）**:
```json
{
  "inquiry_only": false,
  "all_available": true,
  "all_reserved": true,
  "items": [
    {
      "product_name": "りんご",
      "available": true,
      "reserved": true,
      "reserved_qty": 10
    }
  ],
  "message": "りんご10箱の在庫を確保しました"
}
```

**よくある間違い**:
- 1回目の問い合わせで引当してしまう。その後の注文時に在庫がない
- 「前回15箱あると言ったから確保してある」と勘違いする（問い合わせは確保しない）

**関連**: Q10, Q11

### カテゴリE: 特殊なケース

#### Q13. 全体キャンセルの場合、在庫はどうなる？

**状況**: 既に在庫引当済みの注文（りんご10箱、reserved = true）が全体キャンセルになった。

**注文内容**: りんご 10箱（キャンセル指示）
**在庫状態**: 引当済み（誰かのものになっていた）

**正しい判断**:
引当済みの在庫を**解放する**。reserve_inventory の逆操作。そうしないと、他の注文が「なぜ引当できないの？」となります。
- action = "release_reservation"
- その10箱は再び誰でも注文できる状態に戻す

**出力**:
```json
{
  "action": "release_reservation",
  "product_name": "りんご",
  "released_qty": 10,
  "message": "引当を解放しました。在庫が利用可能になります"
}
```

**よくある間違い**:
- キャンセル注文を「ignore」して、引当を解放しない
- 結果：その10箱が誰にも使われない幽霊在庫になる

**関連**: Q1（引当）

#### Q14. 現在注文に商品が追加された場合、在庫確認はどの範囲？

**状況**: 現在の進行中の注文：りんご10箱（既に引当済み）。ここに白菜2ケースが追加される。

**既存注文**: りんご 10箱（all_reserved = true）
**追加商品**: 白菜 2ケース
**在庫状態**: 白菜 50ケース

**正しい判断**:
追加された白菜2ケース**のみ** check_inventory + reserve。りんご10箱は既に引当済みなので、もう一度チェックする必要はありません。
- りんご：スキップ（既に処理済み）
- 白菜：新規チェック + reserve

**出力**:
```json
{
  "action": "add_to_order",
  "existing_items": [
    {"product_name": "りんご", "reserved_qty": 10, "status": "既引当"}
  ],
  "new_items": [
    {
      "product_name": "白菜",
      "available": true,
      "reserved": true,
      "reserved_qty": 2
    }
  ],
  "message": "白菜2ケースの在庫を追加で確保しました"
}
```

**よくある間違い**:
- 既に引当済みのりんごをもう一度チェック。無駄（実害はないけど効率悪い）
- 白菜をチェック忘れ

**関連**: Q1（引当の基本）

---

#### Q15. 同じ商品を2つの顧客が同時に注文して、在庫が1つ分しかない場合は？

**状況**: りんご在庫10箱。顧客A が「りんご10箱」、顧客B が「りんご10箱」をほぼ同時に注文。

**顧客A注文**: りんご 10箱
**顧客B注文**: りんご 10箱
**在庫状態**: りんご 10箱（1つの注文分しかない）

**正しい判断**:
reserve_inventory が成功するのは1つだけ。
- 顧客A: reserve 成功 → all_reserved = true
- 顧客B: reserve 失敗（在庫がない） → all_reserved = false

後からの顧客Bには「在庫不足」で返す。これが「引当」の役割です。
- `all_available` の値は「チェック時点で在庫があるか」
- `all_reserved` の値は「実際に確保できたか」

**出力（顧客A）**:
```json
{
  "all_available": true,
  "all_reserved": true,
  "items": [{"product_name": "りんご", "reserved_qty": 10}],
  "message": "りんご10箱の在庫を確保しました"
}
```

**出力（顧客B）**:
```json
{
  "all_available": true,
  "all_reserved": false,
  "items": [
    {
      "product_name": "りんご",
      "available_qty": 0,
      "requested_qty": 10
    }
  ],
  "status": "要対応",
  "message": "申し訳ございません。りんご10箱は他の注文により確保されています"
}
```

**よくある間違い**:
- 在庫チェック時点でOKだったから、両方OKにしてしまう
- 引当操作をスキップ
- 「どっちに売るのか」という争奪を回避できない

**関連**: Q1（引当の重要性）, Q16（引当失敗）

---

#### Q16. 引当に失敗した場合（競合など）はどうする？

**状況**: check_inventory ではOKだったが、reserve_inventory が失敗（別の注文に取られたなど）。

**注文内容**: バナナ 5kg
**チェック時**: バナナ 5kg 在庫あり（OK）
**引当実行時**: 別の注文に取られて失敗

**正しい判断**:
- `all_available` = true（チェック時点ではあった）
- `all_reserved` = false（でも確保できなかった）

この状況では：
1. **リトライ**: もう一度チェック→引当を試す（他の商品に切り替わったかもしれない）
2. または **要対応**: 担当者に連絡（代替品提案など、人判断が必要）

システムが自動で判断できません。

**出力**:
```json
{
  "all_available": true,
  "all_reserved": false,
  "items": [
    {
      "product_name": "バナナ",
      "available": true,
      "reserved": false,
      "conflict": "Reserve operation failed due to concurrent order"
    }
  ],
  "status": "リトライ or 要対応",
  "message": "申し訳ございません。在庫確認中に他の注文が成立しました。もう一度チェックしています..."
}
```

**よくある間違い**:
- all_available=true だから all_reserved も true にしてしまう（別操作だということを忘れる）
- 引当失敗を無視して「受注確定」にしてしまう

**関連**: Q1（引当の基本）, Q15（競合）

## 5. やってはいけないこと

1. 在庫不足なのに「確保しました」と返す
2. 在庫問い合わせなのに引当を実行する
3. 代替品を勝手に確定する（提案するだけ。確定は顧客の返信待ち）
4. 分納を顧客の了承なしに確定する
5. 引当失敗を無視して受注確定とする
