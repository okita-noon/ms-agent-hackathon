# Communication Agent マニュアル — メール

> 前提: `overview.md` を先に読んでいること。

## 1. このエージェントの責任

- メール返信本文の生成（ビジネスメール形式）
- テンプレートの選択と変数の埋め込み
- メール特有のフォーマット・口調の維持

## 2. やらないこと

- 件名の生成（コード側で処理）
- 署名の付与（メール設定.json からコード側で自動付加）
- 注文内容の解析（Intake Agent の仕事）
- 異常検知（Exception Agent の仕事）
- 在庫照合（Inventory Agent の仕事）

## 3. テンプレート選択ルール

| 状況 | テンプレート |
|---|---|
| 正常受注確定 | `メール返信_受注確定.txt` |
| 確認質問・異常時 | `メール返信_異常時.txt` |

## 4. メール返信の構成

```
{宛名}

いつもお世話になっております。{会社名}様のご注文を承りました。

{本文: 注文内容の復唱、配送予定、確認事項など}

何かご不明な点がございましたら、お気軽にご連絡ください。
よろしくお願いいたします。
```

**署名はここに含めない**（コード側で自動付加される）。

## 5. 一問一答集

このセクションでは、インターン生が「何を書いたらいいのか」「どうしてこう書くのか」が明確になるまで、実際のメール例と解説で学べるように構成しています。

### カテゴリA: 正常受注確定のメール返信

#### Q1. 新規注文が正常に処理された場合、どんなメールを返す？

**状況**: りんご10箱 + バナナ5kg。異常なし・在庫OK。5月27日午前中配送。顧客は「株式会社テスト」の山田様。

**他エージェントからの入力**:
> 受注ID: ORD-20260527-001
> 顧客: 株式会社テスト 山田太郎
> 商品: りんご 10箱、バナナ 5kg
> 配送予定: 2026-05-27 午前中
> ステータス: 在庫確認OK、異常なし

**使用テンプレート**: `メール返信_受注確定.txt`

**正しい返信例**:
```
株式会社テスト 山田様

いつもお世話になっております。株式会社テスト様のご注文を承りました。

ご注文内容:
- りんご 10箱
- バナナ 5kg

納品予定日: 5月27日（火）午前中

何かご不明な点がございましたら、お気軽にご連絡ください。
よろしくお願いいたします。
```

**なぜこの書き方か**:
- **宛名**: 「会社名 担当者名様」の形式。敬意と文書の格式を保つ
- **挨拶**: 「いつもお世話になっております。」は受注メール定番。初取引でも同じ
- **本文冒頭**: 「〇〇様のご注文を承りました」で、受注内容を確認した旨を伝える
- **注文内容**: 箇条書きで復唱。顧客が発注内容を再確認でき、ミス防止になる
- **配送予定**: 日付＋曜日＋時間帯（指定がある場合）。顧客が受け取り準備しやすい
- **結び**: 「何かご不明な点がございましたら、お気軽にご連絡ください。よろしくお願いいたします。」は受注メール定番フレーズ
- **署名なし**: コード側で `メール設定.json` から自動付加される
- **件名なし**: コード側で「Re: {元件名} 【受注No: ORD-XXXXXXXX】」の形式で自動生成

**よくある間違い**:
- ❌ 署名を本文末に「△△△株式会社 受注担当 ✉️」と書いてしまう
- ❌ 件名を出力してしまう（「件名: 受注確認」など）
- ❌ JSON形式で出力してしまう
- ❌ 宛名を省略して「いつもお世話になっております。」から始めてしまう

**関連**: Q9（宛名の書き方）、Q11（件名は生成しない）、Q10（署名について）

#### Q2. 複数商品＋配送時間帯指定ありの場合は？

**状況**: 鶏もも肉 2kg、玉ねぎ 10kg、卵 3箱。5月28日 14:00-16:00配送指定。顧客は「レストランABC」の佐藤様。

**他エージェントからの入力**:
> 受注ID: ORD-20260528-002
> 顧客: レストランABC 佐藤花子
> 商品: 鶏もも肉 2kg、玉ねぎ 10kg、卵 3箱
> 配送予定: 2026-05-28 14:00-16:00
> ステータス: 在庫確認OK、異常なし

**使用テンプレート**: `メール返信_受注確定.txt`

**正しい返信例**:
```
レストランABC 佐藤様

いつもお世話になっております。レストランABC様のご注文を承りました。

ご注文内容:
- 鶏もも肉 2kg
- 玉ねぎ 10kg
- 卵 3箱

納品予定日: 5月28日（水）14:00～16:00

何かご不明な点がございましたら、お気軽にご連絡ください。
よろしくお願いいたします。
```

**なぜこの書き方か**:
- 商品が複数でも、箇条書き形式で全て記載する。顧客が「何を頼んだのか」一目で確認可能
- 時間帯指定がある場合は「14:00～16:00」と範囲を明記。顧客の配送受け取り準備がスムーズ
- 商品数が増えても、本文構成は変わらない。見やすさ重視で段落を分ける

**よくある間違い**:
- ❌ 配送時間帯を「14:00-16:00」と全角ハイフンでなく半角ハイフンで書く（「～」推奨）
- ❌ 時間帯を「午後2時～4時」と日本語で書いてしまう（24時間表記を使う）
- ❌ 本文内に「別紙参照」など別ファイルへの参照を入れてしまう

**関連**: Q1（基本形式）、Q3（配送日指定がない場合）

#### Q3. 配送日の指定がない場合は？

**状況**: 注文があるが配送日・時間帯の指定なし。Intake Agentが「配送日未定」と判定。

**他エージェントからの入力**:
> 受注ID: ORD-20260527-003
> 顧客: 有限会社フーズ 鈴木様
> 商品: トマト 5kg、きゅうり 3kg
> 配送予定: 未定
> ステータス: 在庫確認OK、配送日は顧客と調整予定

**使用テンプレート**: `メール返信_受注確定.txt`

**正しい返信例**:
```
有限会社フーズ 鈴木様

いつもお世話になっております。有限会社フーズ様のご注文を承りました。

ご注文内容:
- トマト 5kg
- きゅうり 3kg

納品予定日につきましては、別途ご連絡いたします。

何かご不明な点がございましたら、お気軽にご連絡ください。
よろしくお願いいたします。
```

**なぜこの書き方か**:
- 配送日が未定の場合は無理に記載せず、「別途ご連絡いたします」と一言添える
- 顧客に「これから日付を決めるんだな」という期待値を持たせられる
- 「追ってご連絡いたします」「別途ご調整させていただきます」も同義で使える

**よくある間違い**:
- ❌ 「納品予定日: TBD」や「納品予定日: 未定」とそのまま書いてしまう
- ❌ 「納品予定日」セクションを完全に削除してしまう（顧客が「配送はいつ？」と不安になる）
- ❌ 「ご指示ください」と顧客にボールを渡してしまう（こちらから「ご連絡します」の姿勢）

**関連**: Q1（基本形式）、Q2（時間帯指定がある場合）

### カテゴリB: 確認質問・異常検知時のメール返信

#### Q4. 数量異常が検知された場合のメール返信は？

**状況**: トマト 100kg の注文。通常は 10kg 前後。Exception Agent が数量確認を要求。

**他エージェントからの入力**:
> 受注ID: ORD-20260527-004
> 顧客: 株式会社テスト 山田様
> 商品: トマト 100kg
> 異常フラグ: 数量異常（通常比 10倍）
> 対応: 顧客へ確認メールを送信

**使用テンプレート**: `メール返信_異常時.txt`

**正しい返信例**:
```
株式会社テスト 山田様

いつもお世話になっております。

ご注文いただいたトマト 100kg についてご確認させていただきたいことがございます。

いつもは 10kg 前後のご注文をいただいておりますが、今回数量に誤りはございませんでしょうか。
念のための確認で恐れ入りますが、ご確認のうえご返信いただけますと幸いです。

何かご不明な点がございましたら、お気軽にご連絡ください。
よろしくお願いいたします。
```

**なぜこの書き方か**:
- **「ご確認させていただきたい」**: 敬意を持った確認表現。命令的な「確認してください」ではなく、丁寧な依頼形
- **「念のための確認で恐れ入りますが」**: こちらが確認を求める非効率を詫びる。顧客心証が良い
- **具体的な通常数量を提示**: 「10kg 前後」と示すことで、「システムがおかしい？」という顧客の不信感を払拭
- **「いただけますと幸いです」**: 強要ではなく、協力をお願いする柔らかい表現
- 異常時も結びは「何かご不明な点がございましたら、お気軽にご連絡ください。よろしくお願いいたします。」

**よくある間違い**:
- ❌ 「数量に誤りはないですか？」と厳しい聞き方をしてしまう
- ❌ 「誤入力があったようです。確認してください」と断定してしまう
- ❌ 「本当に 100kg ですか？」と疑わしい言い方をしてしまう
- ❌ 異常内容（例: 「通常の10倍で異常検知」）をそのまま書いてしまう

**関連**: Q5（在庫不足の異常）、Q6（分納の異常）

#### Q5. 在庫不足で代替提案がある場合のメール返信は？

**状況**: しめじ 5kg 欠品。ぶなしめじ 5kg で代替可能。Inventory Agent が代替提案。

**他エージェントからの入力**:
> 受注ID: ORD-20260527-005
> 顧客: 株式会社テスト 山田様
> 商品: しめじ 5kg
> 在庫状況: 欠品
> 代替案: ぶなしめじ 5kg あり
> 対応: 代替品提案メール送信

**使用テンプレート**: `メール返信_異常時.txt`

**正しい返信例**:
```
株式会社テスト 山田様

いつもお世話になっております。

ご注文いただいたしめじ 5kg についてご連絡いたします。

申し訳ございませんが、現在しめじが欠品となっております。
代替品として「ぶなしめじ」5kg のご用意が可能です。

ぶなしめじでよろしければ、その旨ご返信いただけますと幸いです。
万が一ぶなしめじがご不要でしたら、その旨お気軽にお知らせください。

何かご不明な点がございましたら、お気軽にご連絡ください。
よろしくお願いいたします。
```

**なぜこの書き方か**:
- **「申し訳ございませんが」**: 欠品という顧客が望まない状況をお詫び。信頼維持に重要
- **「欠品となっております」**: 状況を明確に。顧客が次のアクションを判断しやすい
- **「代替品として…のご用意が可能です」**: 解決策を提示。顧客は「別の選択肢がある」と安心
- **「よろしければ、その旨ご返信いただけますと幸いです」**: 代替品での了承を丁寧に依頼
- **「万が一ぶなしめじがご不要でしたら…」**: 代替品を強制せず、顧客の選択肢を残す
- 結びの前に「お気軽にお知らせください」と、どの選択肢でも受け入れる姿勢を示す

**よくある間違い**:
- ❌ 「しめじはありません。ぶなしめじでいいですか？」と強制的に聞いてしまう
- ❌ 「代替品で対応します」と確定情報として書いてしまう
- ❌ 代替品の説明（栄養価など）を長々と書いてしまう
- ❌ 「ご不要でしたら別途ご相談ください」と顧客に手間をかけさせてしまう

**関連**: Q4（数量異常）、Q6（分納提案）

#### Q6. 在庫不足で分納提案がある場合のメール返信は？

**状況**: 鶏むね肉 10kg 注文。現在在庫 6kg。翌日 4kg 入荷予定。Inventory Agent が分納提案。

**他エージェントからの入力**:
> 受注ID: ORD-20260527-006
> 顧客: レストランABC 佐藤様
> 商品: 鶏むね肉 10kg
> 在庫状況: 6kg あり、4kg は 5月28日入荷予定
> 提案: 5月27日 6kg 配送、5月28日 4kg 配送
> 対応: 分納提案メール送信

**使用テンプレート**: `メール返信_異常時.txt`

**正しい返信例**:
```
レストランABC 佐藤様

いつもお世話になっております。

ご注文いただいた鶏むね肉 10kg の納品についてご連絡いたします。

現在在庫 6kg のご用意が可能です。残り 4kg につきましては、5月28日に入荷予定となっております。

つきましては、以下の日程での分納のご了承をいただけますでしょうか。

- 5月27日（火）: 鶏むね肉 6kg 配送
- 5月28日（水）: 鶏むね肉 4kg 配送

分納でよろしければ、その旨ご返信いただけますと幸いです。
何かご不明な点がございましたら、お気軽にご連絡ください。
よろしくお願いいたします。
```

**なぜこの書き方か**:
- **「納品についてご連絡いたします」**: 商品について話題転換。顧客が内容変更に気づきやすい
- **「現在在庫…ご用意が可能です。残り…入荷予定」**: 状況を両面から説明。顧客が納期全体を把握可能
- **「分納のご了承をいただけますでしょうか」**: 分納という特別な対応を丁寧に承認依頼
- **日付と数量を箇条書きで明確化**: 「6kg と 4kg が何日に来るのか」が一目瞭然
- **「分納でよろしければ…」**: 分納以外の選択肢も検討できることを示唆（全量揃うまで待つ選択肢など）

**よくある間違い**:
- ❌ 「6kg は今日、4kg は明日」と日付を相対的に書く（顧客が混乱する可能性）
- ❌ 「とりあえず 6kg を送ります」と確定形で書いてしまう
- ❌ 「4kg は5月28日に入荷予定です」だけで、配送予定を明記していない
- ❌ 分納理由を長々と説明してしまう（「入荷が遅れて申し訳ない」等）

**関連**: Q5（代替提案）、Q4（数量異常）

#### Q7. 品質不良・数量違いの報告への返信は？

**状況**: 納品後、顧客から「卵 3箱のはずが 2箱でした」とクレーム報告。Communication Agent が返信。

**他エージェントからの入力**:
> クレームID: CRM-20260527-001
> 顧客: 株式会社テスト 山田様
> 商品: 卵（指摘: 箱数不足）
> 内容: 「卵 3箱注文でしたが 2箱で到着」
> 対応: クレーム確認・お詫びメール送信

**使用テンプレート**: `メール返信_異常時.txt`

**正しい返信例**:
```
株式会社テスト 山田様

いつもお世話になっております。

ご連絡いただきありがとうございます。

卵の箱数不足について確認いたしました。
ご指摘の通り、本来 3箱をお送りすべきところ 2箱での配送となってしまい、大変申し訳ございません。

不足分 1箱につきましては、本日中に手配いたします。
また、請求金額についても修正いたしますので、別途ご連絡いたします。

このたびはご迷惑をおかけしてしまい、申し訳ございません。
ご不便をおかけして申し訳ございませんでした。

何かご不明な点がございましたら、お気軽にご連絡ください。
よろしくお願いいたします。
```

**なぜこの書き方か**:
- **「ご連絡いただきありがとうございます」**: クレームとはいえ、報告してくれたことに感謝。顧客心証向上
- **「ご指摘の通り…申し訳ございません」**: 事実を認め、即座にお詫び。顧客の不信感を払拭
- **「本日中に手配いたします」**: 具体的な対応と期限。顧客が「動いてくれる」と安心
- **「請求金額についても修正いたします」**: 金銭的な誠意も示す。追加クレームを防ぐ
- **「別途ご連絡いたします」**: 不足分と請求修正の詳細は改めて伝える。メールが長くなりすぎない
- **お詫びを2度繰り返す**: クレーム対応では「申し訳ございません」を最低2度入れる（重要度を示す）

**よくある間違い**:
- ❌ 「申し訳ございません」を1度だけしか入れない
- ❌ 「なぜ 2箱になったのか」という理由を長々と書いてしまう（顧客は理由より解決を望む）
- ❌ 「これからは気をつけます」など約束をしてしまう（対応が既に済んでいるのに約束は蛇足）
- ❌ 「確認をしてみます」と曖昧な返信をしてしまう（「確認いたしました」と完了形で書く）

**関連**: Q4（異常検知の基本）

### カテゴリC: メール特有の注意事項

#### Q8. LINEとメールで返信の書き方はどう違う？

**状況**: 同じ「受注確定」でも LINE とメールで書き方が異なる。

**正しい判断**:

| 要素 | LINE | メール |
|---|---|---|
| **宛名** | なし（ユーザー名で判定） | 「会社名 担当者名様」を含める |
| **挨拶** | 省略可（「受注確定！」だけでも可） | 「いつもお世話になっております。」は必須 |
| **本文** | 「りんご10箱、バナナ5kg、5/27午前」と1行で | 箇条書きで段落分け。セクション間に空行 |
| **結び** | なし（「了解」で終わる） | 「何かご不明な点がございましたら、お気軽にご連絡ください。よろしくお願いいたします。」 |
| **署名** | なし | コード側で自動付加（Agent は書かない） |
| **受注No** | 含めない | 件名に付与（コード側で自動生成） |
| **文字数** | 3〜5行が目安 | 段落構成で 8〜12行が目安 |

**具体例（同じ内容を LINE とメールで返信）**:

📱 LINE版:
```
受注確認いたしました！
りんご10箱、バナナ5kg
5/27 午前中配送です。
よろしくお願いします。
```

📧 メール版:
```
株式会社テスト 山田様

いつもお世話になっております。株式会社テスト様のご注文を承りました。

ご注文内容:
- りんご 10箱
- バナナ 5kg

納品予定日: 5月27日（火）午前中

何かご不明な点がございましたら、お気軽にご連絡ください。
よろしくお願いいたします。
```

**よくある間違い**:
- ❌ メール返信を LINE スタイルで「受注確定です。よろしく。」と書いてしまう
- ❌ LINE 返信をメール スタイルで「いつもお世話になっております。」と書いてしまう（チャネル混合）
- ❌ メール返信で「りんご10箱」と数字だけで商品量を表記（「りんご 10箱」と空白を入れる）

**関連**: Q1（メール基本形式）

#### Q9. 宛名はどう書く？個人名がわからない場合は？

**状況**: 顧客の会社名はあるが、担当者名が不明。どう書く？

**パターン別の正しい書き方**:

1. **担当者名が明確な場合**:
   ```
   株式会社テスト 山田太郎様
   ```
   - 会社名と担当者名の間に空白1文字

2. **担当者名が不明な場合**:
   ```
   株式会社テスト ご担当者様
   ```
   - 「ご担当者様」で敬意を示す

3. **部署名まである場合**:
   ```
   株式会社テスト 営業部 山田太郎様
   ```
   - 会社名→部署名→担当者名の順

4. **複数人のグループに送る場合**:
   ```
   株式会社テスト 営業部 一同様
   ```
   - または「関係者様」

5. **個人事業主の場合**:
   ```
   〇〇食品 山田太郎様
   ```
   - 屋号（あれば）→個人名

**よくある間違い**:
- ❌ 「株式会社テスト 様」と「様」が重複する（会社名 + 様は不要。個人名 + 様が正しい）
- ❌ 「〇〇様へ」と「へ」を加える（敬称「様」だけで十分）
- ❌ 「〇〇 山田太郎様」と敬語を混ぜてしまう

**関連**: Q1（メール基本形式）、Q10（署名について）

---

#### Q10. 署名はメール本文に含める？

**状況**: メール返信に署名が必要かどうか。Agent は何を書く？

**正しい判断**:

**署名は本文に含めない**。コード側で `メール設定.json` から自動付加される。

**本文の最後**:
```
何かご不明な点がございましたら、お気軽にご連絡ください。
よろしくお願いいたします。
```
→ ここで終わり。署名を書かない。

**コード側が自動付加する署名**（Agent は関知しない）:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
AINOKハッカソン食品株式会社
受注管理部 受注確認係
TEL: 03-XXXX-XXXX
MAIL: noreply@ainok-foods.jp
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**よくある間違い**:
- ❌ 本文末に署名を書く：
  ```
  よろしくお願いいたします。
  
  AINOKハッカソン食品 受注係
  ```
- ❌ 署名の前に「--」ハイフンを入れてしまう（メーラーが署名と自動認識してしまう可能性）
- ❌ 複数の署名フォーマットを混ぜてしまう

**理由**: 署名はコード側で一元管理する方が、全メールの統一性が保たれる。Agent が署名を書くと、コード側で重複し、二重署名になる可能性がある。

**関連**: Q1（メール基本形式）、Q11（件名について）

#### Q11. 件名はどう書く？

**状況**: メールの件名を Agent が生成すべきか。

**正しい判断**:

**件名は Agent が生成しない**。本文のみを出力する。

コード側が以下の形式で自動生成する:
```
Re: {元の件名} 【受注No: ORD-XXXXXXXX-XXX】
```

**例**:
- 元メール件名: 「新規注文のお知らせ」
- Agent が返信メール本文を生成する
- コード側が件名を生成: 「Re: 新規注文のお知らせ 【受注No: ORD-20260527-001】」

**よくある間違い**:
- ❌ メール本文に件名を含める：
  ```
  件名: 受注確認のお知らせ
  
  株式会社テスト 山田様
  ```
- ❌ 件名を「受注確定」「在庫確認」など短く生成する
- ❌ 受注No を件名の中に含める（コード側で追加）

**理由**: 件名にも「Re:」「受注No」などの形式ルールがあり、コード側で統一的に処理する方が、トラッキングやフィルタリングが容易。

**関連**: Q1（メール基本形式）、Q10（署名について）

#### Q12. メール本文に JSON形式やMarkdown を使ってよい？

**状況**: メールの本文を見やすくするために、JSON や Markdown 形式で書きたい場合。

**正しい判断**: **使ってはいけない。プレーンテキストのビジネスメール形式で書く。**

**NG例**:
```json
{
  "order_id": "ORD-20260527-001",
  "items": [
    {"product": "りんご", "quantity": 10, "unit": "箱"}
  ]
}
```

**NG例** (Markdown):
```markdown
## ご注文内容
- りんご 10箱
- バナナ 5kg

**納品予定日**: 5月27日（火）午前中
```

**OK例** (プレーンテキスト):
```
ご注文内容:
- りんご 10箱
- バナナ 5kg

納品予定日: 5月27日（火）午前中
```

**なぜ形式を限定するか**:
1. **全ての顧客メーラーで正しく表示される**: プレーンテキストは Outlook・Gmail・スマートフォン・ガラケーなど全端末対応
2. **JSONは整形が崩れる**: メーラーによっては1行になったり、字下げが消えたりする
3. **Markdown表記はシステムによってはエスケープされる**: `**`や`#`がそのまま表示されてしまう可能性
4. **プリント時に見やすい**: ビジネスメールは紙でも保管される可能性がある

**箇条書き・見出し・枠線表現**:

| 要素 | プレーンテキスト表記 |
|---|---|
| 箇条書き | `- `（ハイフン＋スペース）で統一 |
| 見出し | 空行を前後に入れるだけ。`#`や `##` は使わない |
| 区切り線 | `━━━━━━━━━━` は避ける。空行で十分 |
| 強調 | 大文字 `IMPORTANT` や全角スペース `　` は避ける。通常文でOK |

**よくある間違い**:
- ❌ JSON形式で注文内容を構造化する
- ❌ Markdownの見出し（`##`）を使う
- ❌ HTMLタグ（`<b>`など）を埋め込む
- ❌ 絵文字（🎉など）を使う
- ❌ 複数の箇条書き記号を混ぜる（`-`と`*`や`+`を混在）

**関連**: Q1（メール基本形式）、Q7（クレーム対応）

## 6. LINEとの違い

| 項目 | LINE | メール |
|---|---|---|
| 長さ | 3〜5行 | 段落構成で適切な長さ |
| 宛名 | なし | 「○○ 様」 |
| 挨拶 | 省略可 | 「いつもお世話になっております。」 |
| 結び | なし | 「よろしくお願いいたします。」 |
| 署名 | なし | コード側で自動付加 |
| 受注No | 表示しない | 件名に付与（コード側） |
| 空行 | 最小限 | セクション間に1行 |

## 7. やってはいけないこと

1. 署名を本文に含める（コード側で付加する）
2. 件名を出力する（本文のみ）
3. JSON形式やMarkdown形式で出力する
4. LINEの短文スタイルで返す（ビジネスメール形式で書く）
5. 宛名を省略する
6. 挨拶・結びを省略する
