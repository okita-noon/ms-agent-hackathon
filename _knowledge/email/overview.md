# メール受発注 業務ナレッジ — 全エージェント共通

> すべてのエージェントはこのドキュメントを最初に読むこと。

## 1. 業務の大前提

- 食品卸（売り手）が、飲食店（買い手）からの注文をメールで受ける業務である
- メールはLINEと比べて情報量が多く、1通で複数商品・納品条件をまとめて指定するケースが多い
- 1顧客1オープン注文ルールはLINEと同じ
- メールの返信はビジネスメール形式で構成する（LINEの短文とは異なる）

## 2. メールチャネルの特性

| 項目 | LINE | メール |
|---|---|---|
| 情報量 | 短文・省略が多い | 比較的まとまった情報 |
| 会話の頻度 | 高頻度の往復 | 1〜2往復で完結が多い |
| 口調 | カジュアル寄り | ビジネスメール |
| 受注Noの表示 | しない | する（件名・本文に付与） |
| 署名 | なし | あり（メール設定.json から付加） |
| 返信フォーマット | テンプレート短文 | ビジネスメール構成（宛名→挨拶→本文→結び→署名） |

## 3. メール返信の構成

メール返信は以下の構成で作成する。

```
1. 宛名: 「○○様」
2. 挨拶: 「いつもお世話になっております。○○様のご注文を承りました。」
3. 本文: 注文内容・確認事項・配送予定など
4. 結び: 「何かご不明な点がございましたら、お気軽にご連絡ください。」
5. 署名: メール設定.json から自動付加（コード側で処理。Agentは署名を書かない）
```

### 件名ルール

- 受注確定時: `Re: {元の件名} 【受注No: ORD-XXXXXXXX-XXX】`
- 確認質問時: `Re: {元の件名}`
- 件名はコード側で付与する。Agentは本文のみ生成する

## 4. メールのテンプレート

| テンプレート | 用途 | 主要変数 |
|---|---|---|
| `メール返信_受注確定.txt` | 正常受注時 | `${customer_name}`, `${company_name}`, `${order_items}`, `${delivery_estimate}` |
| `メール返信_異常時.txt` | 確認質問・異常時 | `${customer_name}`, `${company_name}`, `${body}` |

## 5. メール特有の注意事項

1. **署名はAgentが生成しない** — `メール設定.json` からコード側で自動付加される
2. **件名はAgentが生成しない** — コード側で受注No付きの件名を構成する
3. **受注Noはメール返信に含める** — LINEとは異なる
4. **各セクション間に空行を1行入れる** — 読みやすさのため
5. **JSON等の構造データは出力しない** — 本文のみ

## 6. エージェント構成

メールの処理も LINE と同じ4エージェント構成を使う。
違いは主に Communication Agent の返信フォーマットのみ。

- Intake Agent: 注文解析（メール本文からの抽出。LINE版と同じロジック）
- Exception Agent: 異常検知（LINE版と同じ）
- Inventory Agent: 在庫照合（LINE版と同じ）
- Communication Agent: **メール用の返信フォーマットで生成**（→ `communication_manual.md` 参照）
